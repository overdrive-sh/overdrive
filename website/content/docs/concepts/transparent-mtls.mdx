This is the layer that finally puts identity on the wire. The
[built-in CA](/docs/concepts/identity) mints a SPIFFE SVID for every workload;
the [lifecycle reconciler](/docs/concepts/workload-identity-lifecycle) holds it
in-process and drops it on stop. Neither one encrypts a packet. This subsystem
does: it terminates and originates TLS 1.3 on the workload's behalf, presenting
the workload's SVID, so two workloads that opened plain sockets end up talking
ciphertext authenticated to the platform's trust bundle. Both directions run in
`overdrive serve`, and a packet capture proves it — TLS 1.3 records on the wire,
zero plaintext.

<Callout title="v1 scope">
	Transparent mTLS encrypts mesh traffic in both directions for process workloads
	on a single node, and a wire capture proves it. Each workload is reachable at its
	own canonical mesh address; a client that dials that address on a service port is
	captured, handshaked, and delivered to the server over TLS 1.3. Authentication is
	chain-to-bundle — a presented SVID is verified against the trust bundle, and
	pinning a specific expected peer identity is not part of this cut. Workloads dial
	concrete canonical addresses; resolving a service name to one is a separate
	node-local responder. See [Where it stands](#where-it-stands).
</Callout>

## The workload holds nothing

A workload under transparent mTLS holds no certificate and no private key, and it
runs no TLS code. It opens an ordinary TCP socket to an ordinary address. There is
no sidecar to intercept the call and no library to link. This is the same
sidecarless stance the rest of the platform takes (see the
[architecture overview](/docs/concepts/architecture)): the credential's only
holder is the node agent, never the workload.

The node agent owns the identity and does the cryptography. The workload runs
inside its own network namespace, and an nftables TPROXY rule on its link
redirects each outbound connection to a proxy leg the agent owns. The agent runs
the mutual-TLS handshake there, presenting the workload's SVID, and the
kernel encrypts the steady-state byte stream. The workload sees a normal socket
the whole time. A compromised workload cannot turn this off, downgrade it, or
read the key, because it never had the key and the enforcement lives outside its
network namespace and its address space.

## One proxy, both directions

The enforcement mechanism is a single **agent-light L4 proxy**, the same shape
for the outbound (client) and inbound (server) halves. An earlier design split
this into two mechanisms — an in-band kernel path for process workloads and a
separate tap proxy for sealed VMs. Real-kernel spikes settled the question
empirically: one universal proxy is lossless, needs no kernel patch, and has no
engagement race, so the two-mechanism framing was collapsed into one.

"Agent-light" is a precise claim, and it is easy to overstate. It means the agent
runs no TLS cryptography in userspace — the kernel's kTLS engine encrypts and
decrypts every record. It does **not** mean the agent is idle or that every byte
is zero-copy. The two directions across a kTLS leg are not symmetric:

- **Decrypt (the receive side)** is a genuine zero-copy `splice` out of a plain
  kTLS-RX leg. The kernel decrypts on the splice, and the agent issues only the
  splice call — no per-byte copy, no syscall per record.
- **Encrypt (the send side)** is a bounded `read` then `write_all` that copies
  each record's plaintext through a userspace buffer into a kTLS-TX leg, where the
  kernel encrypts it synchronously. This pays a copy and a syscall per record.

The send side is a copy rather than a `splice` for a concrete reason: a `splice`
into a kTLS-TX leg reports success but silently drops records under
`MSG_DONTWAIT`, so the blocking `write_all` is the only primitive that delivers
every record. Naming that loss class is what retired the original in-kernel
redirect design.

## Outbound: capture, resolve, encrypt

The outbound path turns a workload's plain `connect()` into an authenticated,
encrypted stream — or leaves it alone, or refuses it — by deciding each connection
as it happens. It never consults a pre-programmed table of destinations.

1. **Capture.** Each workload runs in its own network namespace. An nftables
   TPROXY rule on the host side of its veth pair redirects the workload's outbound
   connection to an `IP_TRANSPARENT` listener the agent owns. The `connect()`
   returns as if it reached the peer directly.
2. **Recover and resolve.** TPROXY preserves the address the workload dialed; the
   agent reads it back with `getsockname` on the accepted connection — the
   original destination, not the proxy leg's own address — then resolves it
   against a live index of running mesh backends. The resolve returns one of three
   verdicts:
   - **A mesh backend** — enforce mTLS, and continue to the handshake.
   - **Not a recognized mesh peer** — pass the connection through as cleartext.
     Transparent mTLS encrypts mesh traffic; it is not a firewall, and a call to a
     non-mesh address is left alone.
   - **A mesh peer with no reachable backend** — drop the connection. The agent
     never falls back to cleartext for traffic that should be encrypted.
3. **Handshake.** On a mesh verdict, the agent runs a rustls **TLS 1.3 client**
   handshake to the backend's agent, presenting the workload's SVID as the
   client certificate.
4. **Encrypt.** The negotiated session keys install into kTLS on the peer-facing
   leg. Forward traffic — the workload's plaintext — is copied into the kTLS-TX leg
   and encrypted in the kernel. Return traffic is spliced zero-copy out of the
   kTLS-RX leg back to the workload.

The workload's identity rides the handshake. The peer verifies the presented SVID
against the shared trust bundle, so the connection is authenticated to a platform
identity, not merely encrypted.

Deciding per connection from a live index — rather than programming a
per-destination map ahead of time — is what makes the address the workload dials
the only input the agent needs. The agent lists the cluster's running backends
when it starts and watches the change stream after, re-listing if it ever falls
behind. A backend becomes resolvable the moment it is running and stops resolving
when it goes away.

## Inbound: intercept, verify, deliver

The inbound half mirrors the outbound one, with the agent acting as the TLS
server. A TPROXY rule and an `IP_TRANSPARENT` listener intercept traffic aimed at
the server workload; the agent recovers the original destination with
`getsockname`, runs a server-side mutual-TLS handshake that **requires and
verifies** a client certificate against the trust bundle, arms kTLS-RX, and
splices byte-exact plaintext to the identity-unaware server. A client that cannot
present a valid chain never reaches the workload.

The match key is the workload's **canonical mesh address**. Each allocation gets
its own address on the mesh — its routable identity — and the agent advertises that
address as the workload's backend. When the allocation starts, the agent installs
one capture rule per declared service port, each keyed on the canonical address and
that port. A client that dials `canonical-address:service-port` lands on the rule,
the agent's server leg terminates the handshake, and the byte stream splices through
to the workload. The client dials the concrete address directly; resolving a service
name to it is a separate node-local responder.

## Fail closed, never cleartext

Encryption that silently degrades to plaintext is worse than none, because it
reads as protection while providing none. Overdrive refuses that trade at three
points.

At boot, `overdrive serve` constructs the mTLS dataplane and probes it — both the
enforcement path and the resolve index — **after** the identity manager is live. A
failed probe emits a `health.startup.refused` event and the node refuses to start.
It does not boot into a degraded, cleartext mode.

Per allocation, the agent provisions the workload's network namespace and TPROXY
capture before the workload starts. If any of that fails — the namespace and veth
pair, the egress rule, or the proxy listeners — the allocation is driven to a
terminal `Failed` state rather than left `Running` with unencrypted traffic.

After a restart, the agent rebuilds its picture of which namespaces and capture
rules belong to which allocations, re-adopting the ones that survived and sweeping
the ones whose workloads are gone. A namespace or rule it cannot account for fails
the boot rather than leaving stale capture in place. The boot path refuses the
node; the per-allocation path refuses the workload; the restart path refuses to
run on a picture it cannot trust. None falls back to cleartext.

## How we know

The honest proof is a kernel side-effect, not a log line. A raw `AF_PACKET`
capture on the peer-facing wire during an outbound deploy shows TLS 1.3
application-data records — `0x17` content type — in both directions, and a scan of
the same capture finds zero plaintext bytes from the workload's payload. The
egress capture rule is visible in `nft list ruleset`, the workload's network
namespace in `ip netns identify`, and the kernel's TLS upper-layer protocol state
in `ss -tie`. The agent's recovery of the dialed address is checked against the
address the workload actually used. The inbound path is proven the same way: a
deploy of two mesh workloads runs a real client handshake to the server's canonical
address, and it completes only because the server-side capture rule routes the
connection to the agent's leg — remove the rule and the handshake fails. The
handshake-rejection and boot-refusal cases
are proven by their negatives: a client without a valid chain is refused, and a
node with an injected probe fault refuses to boot.

These run as real-kernel integration tests under the project's Lima harness — the
canonical path for anything that touches kTLS, network namespaces, or nftables
capture.

## Where it stands

Transparent mTLS encrypts mesh traffic in both directions, wired into `serve`:

- **Both directions in a deploy.** A process workload's outbound connection is
  captured on egress, resolved per connection to its mesh backend, handshaked with
  the workload's SVID, and encrypted with TLS 1.3 over kTLS. A connection inbound to
  a workload's canonical mesh address is captured by a per-port rule the agent
  installs when the allocation starts, handshaked server-side, and spliced to the
  workload. Each workload runs in its own network namespace. Boot-time,
  per-allocation, and restart-recovery enforcement all fail closed.
- **Authentication is chain-to-bundle.** A presented SVID is verified against the
  trust bundle. The caller does not pin which specific peer identity it expects.
- **Workloads dial concrete addresses.** The resolve returns a running backend's
  canonical address, not a virtual service IP, and a client dials that address
  directly. Resolving a service name to it is a separate node-local responder.
- **Process workloads, single node.** Sealed-VM and unikernel workloads, and the
  multi-node mesh, are outside this scope.
- **No operator verb.** Encryption is automatic. There is no command to enable or
  disable it; `overdrive deploy <SPEC>` is the only workload verb, and a deployed
  mesh workload is encrypted without further configuration.
