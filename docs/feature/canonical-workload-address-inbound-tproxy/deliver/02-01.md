## Design fidelity (CLAUDE.md "never invent API surface"): **clean**

- `RunningAllocSet.running: BTreeMap<AllocationId, Option<Ipv4Addr>>` is the
  **exact** shape D-BLOCKER2 / feature-delta §"Hydrate path" names — not a near
  miss, not a HashMap, not a bespoke struct.
- Advertise = `workload_addr.unwrap_or(self.host_ipv4)` — exactly D-B2 ("`Some`
  → `workload_addr:port`, `None` → `host_ipv4:port`, fallback UNCHANGED").
- `vip` field on `ServiceBackendRow` **unchanged** (both test arms assert it) —
  honours the "dialable-VIP is #61 territory" boundary.
- No new public surface invented: `empty_for_workload` (`:192`) and
  `ProjectedListener` (`:107`) pre-date this step (#187 rename);
  `project_service_listen_ports` (`:1144`) is from 01-02; the only test helper,
  `write_alloc_status_with_addr`, is a `#[cfg(test)]`-local fn, not API surface.
- Reads the **materialized** addr off the V2 row; does **not** recompute from
  `NetSlot` — exactly the D-BLOCKER2 "persist the join, three readers" contract.

This is the discipline CLAUDE.md's precedent section exists to enforce, and the
crafter held it.

## issue (blocking): criterion 6 — mutation gate has no passing evidence

The step's sixth acceptance criterion is *"Mutation gate ≥80% on the bridge
advertise addr-source selection and the hydrate_actual map population."* Both
named sites are real, diff-mutable production logic introduced by this commit:

- `backend_discovery_bridge.rs` — `workload_addr.unwrap_or(self.host_ipv4)`
  (overdrive-core);
- `reconciler_runtime.rs` — `.map(\|r\| (r.alloc_id, r.workload_addr))`
  (overdrive-control-plane).

The *killing* tests are present and look adequate (the two S-BRIDGE arms use
distinct addrs `10.99.0.6` vs `10.0.0.5`; the hydrate test asserts the map value
on both a `Some` and a `None` alloc). **But the gate was never run** — there is
no on-disk artifact, no DES MUTATION event, no re-run commit. The ≥80%
kill-rate claim is therefore *unsubstantiated*, identical to the 01-02 finding
(`01-02.md` §"C6 mutation gate"), which was treated as blocking and resolved by
the orchestrator re-running at HEAD.

## Non-blocking findings

### suggestion (non-blocking): S-BRIDGE doc-comment over-promises a proptest

`canonical_address_bridge_advertise.rs:20-22` says *"Mandate 9: … PBT-eligible
(proptest over `{Some(addr) | None} x listener_port`)"* but the file ships only
the two `@example` `#[test]` pins — no proptest. The criteria (Some arm / None
arm) are fully met by the example pins, and the feature's property coverage is
carried by S-PORTSET, so this is not a gap in behaviour — but a doc comment
describing a proptest that isn't there will mislead the next reader. Either
add the `{Some|None} × port` proptest or soften the comment to "PBT-eligible;
shipped as `@example` pins (property coverage lives in S-PORTSET)."

### thought (non-blocking): S-PORTSET equality is one-source by construction

Both `capture_port_set` and `advertise_port_set` bottom out in the same
`svc.listeners`, so the equality is structurally close to tautological — a
mutation applied symmetrically to *both* projections would not be caught. The
file is honest about this (`:19-33`) and the non-triviality was demonstrated via
a falsification probe (advertise drops one listener → shrinks to `[1,2]` → fails),
which is the right mitigation for a two-readers-agree property. No action
required; flagged so a future edit doesn't mistake the test for a stronger
guarantee than "the two projections agree given one source."

### praise / process note: out-of-scope fallout correctly surfaced

The `BTreeSet→BTreeMap` widening of the **public** `RunningAllocSet.running`
forced mechanical `.insert(id)` → `.insert(id, None)` edits in three files
outside `implementation_scope` (two `overdrive-sim` invariant fixtures + one
acceptance test). These are genuinely zero-behaviour (`None` preserves the
existing host-fallback semantics the fixtures assert) and unavoidable fallout of
a public-type change — not new scope. The crafter surfaced them explicitly in
the commit body as a scope blocker to the orchestrator, which is exactly the
right move. No issue.
