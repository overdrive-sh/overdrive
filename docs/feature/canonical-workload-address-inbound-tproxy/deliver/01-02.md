## Non-blocking findings

### suggestion (non-blocking): single-source the listener-port projection — `liveness_restart_spec` hand-inlines it

`project_service_listen_ports` exists precisely to be the *one source* for the
declared listener-port set (D-BLOCKER1, "one source / two readers"). Its own
rustdoc warns: *"if a second path derived the port set from anywhere else, the
S-PORTSET equality property (finalized 02-01) would break."*

There is now a second path. `liveness_restart_spec`
(`reconciler_runtime.rs:2973`) populates the field with a hand-inlined copy of
the projection's Service-arm body rather than calling the shared function:

```rust
// project_service_listen_ports (Service arm):
svc.listeners.iter().map(|l| l.port).collect()
// liveness_restart_spec:
service_ports: spec.listeners.iter().map(|l| l.port).collect(),
```

**This is correct today** — `spec: &ServiceV1`, so `spec.listeners` is the
*identical* source field (`ServiceV1.listeners[].port`), and the property holds.
I want to be clear this was the *right call to populate it at all*: defaulting
this site to `Vec::new()` (as the mechanical field-addition pass did elsewhere)
would have silently dropped inbound capture for any liveness-restarted Service —
the crafter correctly recognized this site feeds a real restart spec into
`start_alloc` (03-01) and must carry the real ports.

The residual concern is **drift, not current correctness**: two byte-identical
copies of the projection logic. If `project_service_listen_ports` later gains a
filter (TCP-only), a dedup, or a sort, this copy diverges silently and S-PORTSET
breaks on the restart path only — the hardest case to catch. Recommend a single
`ServiceV1::listen_ports(&self) -> Vec<NonZeroU16>` method that both
`project_service_listen_ports` (Service arm) and `liveness_restart_spec` call,
so the logic literally exists once. Low effort; high alignment with the
step's own single-source thesis.

### nitpick (non-blocking): the "declaration order" claim is under-tested

Criterion 3 and the rustdoc both assert the Service projects its ports *"in
declaration order."* Neither the unit example (`[8080, 9090]`) nor the proptest
(`btree_set` → ascending) can distinguish *order-preserving* from *sorting* — both
inputs are already ascending, so a `project_*` that sorted its output would pass
identically. To actually pin "declaration order," use a descending or shuffled
listener list in the example arm (e.g. assert `[9090, 8080]` projects to
`[9090, 8080]`, not `[8080, 9090]`). Purely a coverage-precision point; the
behaviour is correct, the assertion just doesn't prove the specific claim.

### praise: the exit-observer initial-population / forward-carry split is well-judged and documented

Criterion 5 reads *"Exit-observer ... copies `spec.workload_addr` into the
written `AllocStatusRowV2.workload_addr`,"* but the exit observer is a *successor*
writer (Terminated/Failed) and has no fresh `spec` — only the prior row and the
exit event. Rather than improvise a way to thread a spec into it, the
implementation correctly split the concern: the **initial population** happens at
the action-shim `StartAllocation`/`RestartAllocation` Running write
(`row.workload_addr = spec.workload_addr`, gated on `state == Running`), and the
exit observer **forward-carries** `prior.workload_addr` (`exit_observer.rs:600`),
exactly mirroring how `started_at`/`kind` already flow. This is the only sound
reading of the criterion, it's the observed-input semantics the design asked
for (not a recompute), and every site carries a comment explaining the split.
Good judgment at an underspecified seam.

### praise: strict design fidelity — no invented API surface

Against CLAUDE.md *"implement to the design — never invent API surface,"* this
step is clean. Exactly the two `AllocationSpec` fields, one projection function,
and the named injection assignments — **no** new spec type, **no** new
`ObservationStore` method (Obligation #2b honoured: `read_job` was widened to
return the projection in its existing tuple, threaded into the existing
`WorkloadLifecycleState` at the same site as `probe_descriptors`), **no** builder
override (the dependency is set at the C3 site, not defaulted). The one
structural change beyond the literal field-adds — extracting
`hydrate_workload_lifecycle_actual` out of `hydrate_actual` — is a legitimate
in-scope `clippy::too_many_lines` fix with a cited precedent
(`hydrate_svid_actual_held`), not invented surface. The `project_service_listen_ports`
function mirrors `project_probe_descriptors` one-for-one and its tests build the
`Service` arm through the real `ServiceV1::from_submit` parser path, so the
generator can't mask a `svc.listeners` shape mismatch — the producer side of
S-PORTSET is structurally honest.
