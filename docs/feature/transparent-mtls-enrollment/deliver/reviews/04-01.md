## Blocking issues

### `issue (blocking):` AC14 (the merged C3 half) has no acceptance test, and one of its named behaviors is unimplemented

The roadmap merged former 04-03 into this step; AC14 specifies a Tier-3
acceptance for the C3 action-shim seam with four sub-claims:

1. real exec alloc reaching Running has its netns+veth provisioned BEFORE spawn,
2. the workload **lands in `ovd-ns-<slot>`** (`ip netns identify`),
3. on terminal the netns is torn down (teardown-then-release),
4. **provision-failure → Failed row, never Running-with-no-netns (mirrors
   `fail_closed_on_mtls_install`).**

The roadmap even names the file: `tests/integration/alloc_netns_lifecycle.rs`.
**That file does not exist, and no test exercises the seam.** A workspace grep
for `provision_and_inject_netns | teardown_and_release_netns | NetSlotExhausted |
WorkloadNetnsProvision` across all `tests/` returns only `streaming_submit.rs`,
which merely threads the now-required `net_slot_allocator` param into `dispatch`
with `mtls_worker: None` — i.e. the seam's **active** path (`mtls_worker.is_some()`)
is never executed by any test. The pieces are tested in isolation
(`workload_netns_provision` = provisioner free-fn; `netns_entry` = `Driver::start`
setns against a hand-created netns; `start_alloc_installs` = the worker install
with `netns: None`, `host_veth` passed manually), but the integration the AC asks
for — StartAllocation → provision → land → terminal teardown — is unproven.

The execution log's RED_UNIT note ("the swap is wiring exercised by the Tier-3
AT") does not hold: the Tier-3 AT it refers to (`start_alloc_installs_both_tproxy`)
sets `netns: None` and supplies `host_veth` directly, so it touches **none** of
the netns seam. For a change that touches the composition root + the dispatch arm
that decides Running-vs-Failed and lands a real workload, the project's own rule
("a `--no-run` gate is green even when every fixture refuses at boot"; integration
coverage is load-bearing) applies squarely.

**Worse, sub-claim 4 is not implemented.** `provision_and_inject_netns(...)?`
(action_shim/mod.rs:994, :1156) bubbles a bare `ShimError::WorkloadNetnsProvision`
/ `NetSlotExhausted` — it does **not** write a superseding `Failed` row the way
`fail_closed_on_mtls_install` does. The safety invariant *"never Running-with-no-
netns"* IS met (provision precedes `driver.start`, so a failure aborts before
Running). But "→ Failed row, mirrors `fail_closed_on_mtls_install`" is false: a
persistent provision failure (slot exhaustion, EPERM removing/creating the netns)
returns `Err` from `dispatch`, leaves the alloc in its prior `Pending` state, and
the reconciler re-emits StartAllocation next tick → an indefinite Pending retry
loop, never a `Failed` terminal.

**Required:** either (a) implement the Failed-row supersede on provision failure
to match AC14 and add the named action-shim Tier-3 AT (lands-in-`ovd-ns-<slot>` +
terminal teardown + provision-failure→Failed), or (b) surface to the orchestrator
that AC14's Failed-row sub-claim is being changed to "abort-to-Pending-and-retry,"
with rationale, and get the AC amended. Silently shipping a divergent behavior
*and* skipping the test is the failure mode CLAUDE.md § "Implement to the
design" exists to prevent. (Slot exhaustion in particular — retry-forever vs.
Failed — is an operator-visible behavior that should be a decision, not an
accident of `?`.)

### `issue (blocking):` AC2 of the outbound-swap AT is a vacuous assertion

`start_alloc_installs_both_tproxy.rs` AC2 claims to prove the deletion is
"observable at the kernel boundary, not just in source" via
`!cgroup_connect4_mtls_program_loaded()` (`bpftool prog show` contains no
`cgroup_connect4_mtls`). It proves nothing:

- The test builds the worker with `SimMtlsEnforcement` and `start_alloc` only
  shells out to `nft`/`ip` + binds loopback listeners. **Nothing in the test
  process ever loads `overdrive_bpf.o`** (or any BPF program). So
  `bpftool prog show` would list no `cgroup_connect4_mtls` regardless of whether
  the program still existed in the ELF — the assertion would have passed
  *identically before the deletion*.
- `cgroup_connect4_mtls_program_loaded()` returns `false` when `bpftool` is
  absent or errors (`Err(_) => false`), so `!loaded` also passes on a toolless
  host.

The deletion IS real — but it is the reviewer's independent ELF section dump
(`no cgroup/connect4_mtls` in the freshly built object) that proves it, not this
AC. This is the recurring vacuous-assertion class
([[project_transparent_mtls_02_02_vacuous_sysctl_assertions]],
[[project_transparent_mtls_03_03_f5_egress_exemption_vacuous]]).

**Required:** make AC2 load the actual `overdrive_bpf.o` and assert the program is
absent from the object's section table (the honest litmus — re-adding the program
to the ELF would then turn it RED), or drop the "observable at the kernel
boundary" claim and assert deletion at the source/ELF level honestly. As written,
AC2 is non-load-bearing and reads as coverage it does not provide.

## Non-blocking issues / suggestions

### `suggestion (non-blocking):` `start_alloc`'s outbound comment documents behavior that does not happen, and the roadmap "outbound = working production path NOW" claim is false

In `mtls_intercept_worker.rs` the outbound accept arm (:535–541, :572–584)
states `accept_outbound_leg` builds `Routed::Outbound { peer }` from the
getsockname-recovered orig-dst and "`enforce` dials the REAL peer." That code
path (:584) is **unreachable in production**: `real_peer` is created `None`
(:414) and never written (`program_declared_peer_redirect` was deleted this
step; the module doc at :51 says the slot is "inert — nothing programs it"), so
the `let Some(peer) = *real_peer.lock() else { … }` at :553 always takes the
`else` → `accept_drop_outbound` → fail-closed-drop. **All leg-F traffic
fail-closed-drops as of 04-01.** Within the same comment block, :542–544 admits
this ("vestigial … ALWAYS `None` … leg-F traffic currently fail-closed-drops
until 04-02"). The two halves contradict each other; a reader cannot tell what
the code does.

Consequently the roadmap implementation_notes / re-plan claim *"outbound =
working production path NOW (getsockname + ServiceBackendsResolve, no #178
source)"* is not realized in code — outbound traffic does not complete until
04-02 wires the resolve consumer. The *install* (rule + listeners + accept loop)
is in scope and present; the *traffic path* is not. Per `.claude/rules/...` §
"No aspirational docs," delete the half-comment that describes the unreachable
getsockname dial, state plainly that 04-01 installs the rule and listeners while
leg-F fail-closed-drops until 04-02, and correct the "working production path
NOW" framing (this is the [[feedback_verify_unproven_claims_against_evidence]]
pattern). Note [[project_transparent_mtls_06x_recurring_defects]] sibling memory:
the stale docs here were flagged as "deferred to 04-02 cleanup," so partial
cleanup is acceptable — but the *new* internal contradiction introduced here
should not ship.

### `suggestion (non-blocking):` stale `with_netns_path` name + docstring in `exec_driver/netns_entry.rs`

AC12 said `ExecDriver::with_netns_path` is DELETED single-cut and the netns_entry
fixtures rewritten onto `spec.netns`. The *body* was rewritten (drives
`Driver::start` with `spec.netns`, :155), but the module docstring (:1–6) and the

### `suggestion (non-blocking):` AC4 of the outbound AT does not actually assert "the shared chain survives"

The `stop_alloc` check (:294–302) is guarded `if let Ok(dump_after_stop)` with
"If the chain itself is gone that's also acceptable." So the test passes whether
the shared chain survives or is razed — the AC4 invariant "per-veth teardown, NOT
raze" is unverified. Tighten to assert the chain still exists after stop (then
that only the per-veth rule is gone), or document why raze is tolerable.

### `nitpick (non-blocking):` two adjacent `Option<String>` fields invite transposition

`AllocationSpec.netns` and `host_veth` are both `Option<String>`; the design
(JOIN-1) pinned `String`-not-newtype with a documented rationale (slot-derived,
no parse surface), so this is not a newtype-rule violation. But two adjacent
same-typed optional name fields are a mild call-site footgun. Non-blocking;
flagged only because the design door is open to a thin `NetnsName`/`HostVethName`
wrapper later if a transposition ever bites.

## Praise

- `praise:` **The single-cut deletion is genuinely clean.** `clippy -D warnings`
  passes workspace-wide and the ground-truth ELF dump confirms `cgroup_connect4_mtls`
  is gone while the look-alike `cgroup/connect4` LB program correctly survives
  (the named false-positive preserved). Net `3426+/5979−` is the correct
  deletion shape (test surface shrank: `mtls_production_activation` −1220,
  `mtls_e2e_helpers` −874, `mtls_dataplane_outbound_install` −482,
  `mtls_composed_walking_skeleton` −346, …). Every hard surface
  (`MtlsCgroupLink`, `MtlsDestKey`, `MtlsAddrPort`, `OutboundAttach`,
  `MtlsBootError::Load`, `program_redirect_dest`) grep-confirmed gone.
- `praise:` **error.rs is exemplary.** `MtlsBootError::Load` removed, `Probe`
  retained with a precise rustdoc explaining the deletion, and the typed-error
  discipline ("never flatten to `Internal(String)`") preserved so the
  composition root can still `matches!` on the boot-failure variant.
- `praise:` **The host_veth channel gap was STOPPED-and-surfaced, not invented.**
  The execution log shows the step blocked twice ("design under-specifies the
  channel carrying `plan.host_veth`"), surfaced to the orchestrator, and resolved
  by a design amendment (JOIN-1/JOIN-6, wave-decisions.md "Amended 2026-06-20")
  before implementation — exactly the CLAUDE.md "never invent API surface /
  STOP and surface the gap" discipline. The `AllocationSpec` fields are
  thoroughly documented with the newtype-vs-`String` rationale.
- `praise:` **C3 teardown ordering is correct.** `teardown_and_release_netns`
  releases the slot only *after* a successful teardown, so a crash between the two
  leaves the slot HELD (the netns still exists and is reclaimable) — never a
  released-but-undestroyed leak. Both halves idempotent and gated symmetric with
  provision.
- `praise:` **The mTLS-install fail-closed path is well-reasoned.**
  `fail_closed_on_mtls_install` fires BEFORE `release_for_exit_emission` precisely
  so the Running-gate/exit-observer watcher is never released for a now-`Failed`
  alloc, and supersedes the committed Running row with a typed cause-class. (The
  contrast with the *provision* path that does NOT do this is the blocking
  finding above.)

## Summary

The mechanism swap and the single-cut deletion are correct, clean, and
ground-truth-verified — the deletion is real at the ELF boundary, the gates
genuinely pass on the kernel, and the false-`COMMIT-PASS` defect did not recur.
Two blocking issues remain: the merged C3 half (former 04-03) shipped without its
named Tier-3 acceptance test and with its "provision-failure → Failed row"
behavior unimplemented (bare `?`, no supersede); and AC2 of the outbound AT is a
vacuous assertion that proves nothing about the deletion it claims to observe.
The supporting findings (contradictory/aspirational outbound comment + false
"working production path NOW" claim; broken intra-doc link to the deleted
`program_declared_peer_redirect`; stale `with_netns_path` test name; unasserted
AC4 chain-survival) are doc-honesty and test-tightening fixes that should land
with the revision.
