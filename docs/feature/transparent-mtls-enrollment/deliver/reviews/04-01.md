## Blocking issues

### `issue (blocking):` AC14 (the merged C3 half) has no acceptance test, and one of its named behaviors is unimplemented

The roadmap merged former 04-03 into this step; AC14 specifies a Tier-3
acceptance for the C3 action-shim seam with four sub-claims:

1. real exec alloc reaching Running has its netns+veth provisioned BEFORE spawn,
2. the workload **lands in `ovd-ns-<slot>`** (`ip netns identify`),
3. on terminal the netns is torn down (teardown-then-release),
4. **provision-failure → Failed row, never Running-with-no-netns (mirrors
   `fail_closed_on_mtls_install`).**

The roadmap even names the file: `tests/integration/alloc_netns_lifecycle.rs`.
**That file does not exist, and no test exercises the seam.** A workspace grep
for `provision_and_inject_netns | teardown_and_release_netns | NetSlotExhausted |
WorkloadNetnsProvision` across all `tests/` returns only `streaming_submit.rs`,
which merely threads the now-required `net_slot_allocator` param into `dispatch`
with `mtls_worker: None` — i.e. the seam's **active** path (`mtls_worker.is_some()`)
is never executed by any test. The pieces are tested in isolation
(`workload_netns_provision` = provisioner free-fn; `netns_entry` = `Driver::start`
setns against a hand-created netns; `start_alloc_installs` = the worker install
with `netns: None`, `host_veth` passed manually), but the integration the AC asks
for — StartAllocation → provision → land → terminal teardown — is unproven.

The execution log's RED_UNIT note ("the swap is wiring exercised by the Tier-3
AT") does not hold: the Tier-3 AT it refers to (`start_alloc_installs_both_tproxy`)
sets `netns: None` and supplies `host_veth` directly, so it touches **none** of
the netns seam. For a change that touches the composition root + the dispatch arm
that decides Running-vs-Failed and lands a real workload, the project's own rule
("a `--no-run` gate is green even when every fixture refuses at boot"; integration
coverage is load-bearing) applies squarely.

**Worse, sub-claim 4 is not implemented.** `provision_and_inject_netns(...)?`
(action_shim/mod.rs:994, :1156) bubbles a bare `ShimError::WorkloadNetnsProvision`
/ `NetSlotExhausted` — it does **not** write a superseding `Failed` row the way
`fail_closed_on_mtls_install` does. The safety invariant *"never Running-with-no-
netns"* IS met (provision precedes `driver.start`, so a failure aborts before
Running). But "→ Failed row, mirrors `fail_closed_on_mtls_install`" is false: a
persistent provision failure (slot exhaustion, EPERM removing/creating the netns)
returns `Err` from `dispatch`, leaves the alloc in its prior `Pending` state, and
the reconciler re-emits StartAllocation next tick → an indefinite Pending retry
loop, never a `Failed` terminal.

**Required:** either (a) implement the Failed-row supersede on provision failure
to match AC14 and add the named action-shim Tier-3 AT (lands-in-`ovd-ns-<slot>` +
terminal teardown + provision-failure→Failed), or (b) surface to the orchestrator
that AC14's Failed-row sub-claim is being changed to "abort-to-Pending-and-retry,"
with rationale, and get the AC amended. Silently shipping a divergent behavior
*and* skipping the test is the failure mode CLAUDE.md § "Implement to the
design" exists to prevent. (Slot exhaustion in particular — retry-forever vs.
Failed — is an operator-visible behavior that should be a decision, not an
accident of `?`.)

### `issue (blocking):` AC2 of the outbound-swap AT is a vacuous assertion

`start_alloc_installs_both_tproxy.rs` AC2 claims to prove the deletion is
"observable at the kernel boundary, not just in source" via
`!cgroup_connect4_mtls_program_loaded()` (`bpftool prog show` contains no
`cgroup_connect4_mtls`). It proves nothing:

- The test builds the worker with `SimMtlsEnforcement` and `start_alloc` only
  shells out to `nft`/`ip` + binds loopback listeners. **Nothing in the test
  process ever loads `overdrive_bpf.o`** (or any BPF program). So
  `bpftool prog show` would list no `cgroup_connect4_mtls` regardless of whether
  the program still existed in the ELF — the assertion would have passed
  *identically before the deletion*.
- `cgroup_connect4_mtls_program_loaded()` returns `false` when `bpftool` is
  absent or errors (`Err(_) => false`), so `!loaded` also passes on a toolless
  host.

The deletion IS real — but it is the reviewer's independent ELF section dump
(`no cgroup/connect4_mtls` in the freshly built object) that proves it, not this
AC. This is the recurring vacuous-assertion class
([[project_transparent_mtls_02_02_vacuous_sysctl_assertions]],
[[project_transparent_mtls_03_03_f5_egress_exemption_vacuous]]).

**Required:** make AC2 load the actual `overdrive_bpf.o` and assert the program is
absent from the object's section table (the honest litmus — re-adding the program
to the ELF would then turn it RED), or drop the "observable at the kernel
boundary" claim and assert deletion at the source/ELF level honestly. As written,
AC2 is non-load-bearing and reads as coverage it does not provide.

In `mtls_intercept_worker.rs` the outbound accept arm (:535–541, :572–584)

Consequently the roadmap implementation_notes / re-plan claim *"outbound =
working production path NOW (getsockname + ServiceBackendsResolve, no #178
