## Findings (priority-ordered, Conventional Comments)

### issue (blocking): B1 — derived veth names overflow IFNAMSIZ (15) for any `AllocationId` longer than 8 chars; the golden test's own example already overflows

**Location:** `veth_provisioner.rs:287-289` (`derive_workload_netns_plan`), docstring 259-261, test `derives_per_alloc_netns_veth_names_and_addresses` (1634-1635).

```rust
host_veth: format!("{WORKLOAD_HOST_VETH_PREFIX}{id}"),   // "ovd-hv-" (7) + id
workload_veth: format!("{WORKLOAD_VETH_PREFIX}{id}"),    // "ovd-wl-" (7) + id
```

`AllocationId` is a `define_label_newtype!` whose only length bound is `LABEL_MAX = 253` (`crates/overdrive-core/src/id.rs:90,96`). Linux interface names are bounded by **IFNAMSIZ = 15 usable chars** (`dev_valid_name` rejects ≥16). The prefix `ovd-hv-` / `ovd-wl-` is 7 chars, leaving **8** for the id. So **any alloc id ≥ 9 chars produces an interface name `ip link add` will reject** with `"name" too long`.

The step's *own golden test* enshrines an over-long name as correct:
```rust
assert_eq!(plan.host_veth, "ovd-hv-payments-0");   // 17 chars > 15  — UNCREATABLE
```
`"payments-0"` is 10 chars; `ovd-hv-payments-0` = 17. 02-02 will `ip link add ovd-hv-payments-0` and fail at the kernel. The derivation IS the SSOT for these names and IS 02-01's deliverable — the defect is in this step, latent only because nothing executes yet.

**The docstring claim is false.** Line 260: *"the alloc id is DNS-label-shaped, so the names are valid iface/netns names by construction."* DNS labels (≤63, and `AllocationId` ≤253) are **not** valid interface names (≤15). The "by construction" claim is exactly backwards.

**The repo already solved this and the crafter ignored the precedent.** `crates/overdrive-dataplane/tests/integration/helpers/mtls_netns_topology.rs:115-127`:
```rust
/// Bounded, deterministic 8-hex iface suffix derived from a topology `tag`, so
/// the `wlp-<suffix>` peer veth name fits IFNAMSIZ (15). ...
/// See `.claude/rules/development.md` § "size a derived id to its grammar's ceiling".
fn iface_suffix(tag: &str) -> String { /* 8-hex hash */ }
```
This is a **direct violation of CLAUDE.md § "One shared length ceiling for label-shaped ids"** — a derived id mapped into a destination grammar must size its ceiling to that grammar (here IFNAMSIZ), and the `WorkflowId::for_correlation` precedent in that rule is the same defect class.

**The latent collision is the deeper hazard.** A naïve "fix" that truncates to 15 chars makes the bug worse, not better: `ovd-hv-payments-0` and `ovd-hv-payments-1` both truncate the 8-char id budget to `payments` → both derive `ovd-hv-payments` → **two allocations share one veth name** → cross-allocation interception/routing. (The netns name `ovd-ns-<alloc>` is *not* IFNAMSIZ-bound — it's a `/var/run/netns/` filename, ≤255 — so only the two veth names are affected.)

**Design-inherited — surface, don't unilaterally patch.** `feature-delta.md:59` pins the literal naming: *"`(ovd-wl-<alloc>` in-netns ↔ `ovd-hv-<alloc>` host-side)"*. ADR-0071 and the feature-delta never mention IFNAMSIZ. So the crafter faithfully implemented the design's name — but per CLAUDE.md ("STOP and surface the gap … never reach for the nearest mechanism that compiles") the IFNAMSIZ conflict and the false "valid by construction" claim should have been raised, not papered over. **Recommended resolution (architect):** adopt the existing `iface_suffix`-style bounded hash for the veth ends (e.g. `ovd-hv-<8hex(alloc)>` = 15) while keeping the human-readable `ovd-ns-<alloc>` netns name; then add a proptest invariant `host_veth.len() <= 15 && workload_veth.len() <= 15` over the full 1..=253 id-length space, and a collision proptest (distinct ids ⇒ distinct names).

### issue (blocking): B2 — the converge model cannot bring the in-netns veth end up, nor `lo` up; a netns provisioned from this plan is non-functional

**Location:** `ObservedWorkloadVeth` (313-344) and `WorkloadVethStep` (354-384) — `SetHostVethUp` exists; there is **no** `SetWorkloadVethUp` and **no** loopback-up step; `ObservedWorkloadVeth` has `host_veth_up` but **no** `workload_veth_up` / `lo_up` fact.

Criterion 2 enumerates only "host-side end up," and the crafter implemented it literally. But a veth pair forwards only when **both** ends are up, and a fresh netns has `lo` **down** — both must be brought up by whoever owns netns+veth setup (the provisioner, per feature-delta Q2). The model here is *structurally incapable* of expressing either, so 02-02 cannot converge them without re-opening 02-01's public types.

Ground-truth, both repo precedents do exactly what this model omits:

- `crates/overdrive-testing/src/netns.rs:4-5, 80-81, 99`: *"brings up `lo` inside it (without `lo up` the in-namespace TCP stack rejects loopback)"* and `ip netns exec <ns> ip link set <iface> up` for the moved end.
- The feature's own `mtls_netns_topology.rs:262, 271`: `ip netns exec <netns> ip link set lo up` **and** `ip netns exec <netns> ip link set <wl_veth> up`.
- `spike/findings-egress-tproxy.md`: the validated topology shows `vethW 10.99.0.2/24 … default via .1` with a live `connect()` egress — which requires the in-netns end up (the spike `[step 6] leg-F accepted … peer=10.99.0.2` proves traffic left the in-netns end).

Without these, the `AddDefaultRoute` step (`default via <gateway> dev <workload_veth>`) is installed against a **down** device and the workload cannot egress — defeating the entire point of Path A (a working per-alloc routing point). The converger's docstring claims it produces "the desired **complete** shape," which is false: the complete shape can't carry a packet.

**Design-inherited — surface.** Criterion 2 and the spike's named prereq list both omit in-netns-end-up / lo-up (they're implicit in "netns+veth" plumbing). Per CLAUDE.md this is a gap to surface to the orchestrator, not to silently ship an incomplete model. **Recommended resolution:** add `workload_veth_up: bool` (+ a netns-`lo`-up fact) to `ObservedWorkloadVeth` and `SetWorkloadVethUp` + a loopback-up `WorkloadVethStep`, ordered after `MoveWorkloadEndIntoNetns`; extend `full_ordered_steps()` and the named proptest's iff-emit clauses to match.

### suggestion (non-blocking): S1 — the per-alloc subnet source (IPAM) is unspecified; collision-freedom across concurrent allocs is unproven

`derive_workload_netns_plan` takes `alloc_subnet: Ipv4Net` (resolving the roadmap's `...`). This is a defensible signature, but **who hands out non-overlapping /30s per allocation is undefined**. `feature-delta.md:559` explicitly defers the VIP allocator (#167, not v1) — and while the per-alloc point-to-point /30 is link-local addressing (distinct from a service VIP), *something* must guarantee two concurrent allocs never get overlapping subnets, or two netns get colliding host-side addresses on the host. For a pure step this is acceptable **if surfaced**; flag it as the load-bearing precondition 02-02 (or a later IPAM step) must satisfy, and pin where it's allocated before 02-02 wires execution.

### suggestion (non-blocking): S2 — `workload_addr` drops the host-netns surface's `Option` honesty and silently collides on a degenerate subnet

The host-netns precedent models "no second usable host" honestly: `VethProvisionPlan.backend_gateway: Option<Ipv4Addr>` is `None` (90-92, 169). The new `WorkloadNetnsPlan.workload_addr: Ipv4Addr` (non-optional) instead **falls back to `network()`** (284) — which for a `/32` equals `host_addr`, producing two ends with the **same address** and a `gateway == host_addr == workload_addr`. The docstring claims the derivation "stays total," but "total" here means "returns a silently-wrong plan" for sub-/30 input. Latent (the allocator admits /30, which always has 2 usable hosts), but the commit message claims to "mirror `derive_veth_plan`'s … addressing precedent" and this mirrors only half of it. **Recommend:** either reject sub-/30 subnets in a fallible constructor (make the invalid state unrepresentable) or add a `/31` boundary case to the derivation proptest so the totality claim is actually exercised.

### question (non-blocking): S3 — `rp_filter_relaxed` collapses three sysctls (one of them per-alloc) into one bool

`rp_filter_relaxed` (342-343) is documented as "relaxed on the host-side (ingress) veth + `all` + `lo`." But `all`/`lo` are host-global while the **per-veth** `net.ipv4.conf.<host_veth>.rp_filter` is per-allocation, defaulting to strict on a freshly-created veth. So a *new* alloc on a host where `all`/`lo` are already relaxed still needs its own host-veth knob relaxed — yet whether `RelaxRpFilter` fires then depends entirely on how 02-02's observer computes this single bool. The contract is at least written, but the lossy model pushes a correctness burden onto the observer (cf. `tx_offload` which correctly models *two* ends as *two* bools). **Question for the architect / 02-02:** should `rp_filter_relaxed` be split (global-relaxed vs per-veth-relaxed), or is the documented "true iff all three" contract sufficient and the observer obligated to AND the per-veth knob?

### nitpick (non-blocking): N1 — `workload_converge_steps` discards `plan` (`let _ = plan;`)

The pure diff keys only on observed facts and never reads `plan` (425), unlike the host-netns `converge_steps` which uses `plan.backend_gateway`. Keeping the param is defensible (mirrors the sibling signature; 02-02's executor needs it; documented), but worth a one-line confirmation it's intentional rather than an oversight, since an unused param invites a future reviewer to "simplify" it away and break the 02-02 call shape.

## Findings (priority-ordered, Conventional Comments)

### issue (blocking): B3 — the netns name `ovd-ns-<alloc>` overflows Linux `NAME_MAX` (255) for any alloc id ≥ 249 chars; this is the *identical pigeonhole/ceiling defect class as B1*, on the one derived name D-TME-12's resolution left embedding the unbounded `AllocationId`

**Location:** `derive_workload_netns_plan` (418: `netns: format!("{WORKLOAD_NETNS_PREFIX}{id}")`); the false-reassurance claims at docstrings 204, 344-345, 382-383, **D-TME-12 SSOT** (`wave-decisions.md`: *"a `/var/run/netns/` filename, ≤255, NOT IFNAMSIZ-bound"*), and `feature-delta.md:60` (*"not IFNAMSIZ-bound"*).

```rust
const WORKLOAD_NETNS_PREFIX: &str = "ovd-ns-";          // 7 chars
netns: format!("{WORKLOAD_NETNS_PREFIX}{id}"),          // 7 + alloc_id (≤253) = up to 260
```

`AllocationId` is `define_label_newtype!` bounded *only* by `LABEL_MAX = 253` (`overdrive-core/src/id.rs:90,96` — no tighter per-type ceiling; verified). A netns name created by `ip netns add NAME` becomes a path component under `/run/netns/` and is bounded by **`NAME_MAX` = 255 bytes** (tmpfs); `open(O_CREAT)` on a 256+-byte component returns **`ENAMETOOLONG`**. `ovd-ns-` (7) + a 253-char alloc id = **260 > 255** → `ip netns add` fails at the kernel.

**This is reachable through the public submission surface — not merely theoretical.** The production alloc-id mint site is `reconcilers/workload_lifecycle.rs:838`:
```rust
let raw = format!("alloc-{}-{}", workload_id.as_str(), attempt);   // then AllocationId::new(raw)
```
`WorkloadId` is itself `LABEL_MAX`=253-bounded with **no tighter name-length validation anywhere** (grepped). A workload submitted with a ~244-char name yields a 253-char `AllocationId` → a 260-char netns name → `ENAMETOOLONG` in 02-02's `ip netns add`. The defect IS in this step — `derive_workload_netns_plan` is the SSOT for the name and takes the full-domain `&AllocationId` — latent only because nothing executes yet (exactly the original B1 shape).

**The "≤255" claim is arithmetically false** — it counts the alloc id's ceiling (253) but omits the 7-char prefix (253 + 7 = 260). This is the same reasoning error the first-cut B1 called out (*"the 'by construction' claim is exactly backwards"*): D-TME-12 fixed the two **veth** axes against IFNAMSIZ but reasoned about the **netns** axis as "safe because ≤255" without doing the `+7` arithmetic, leaving the third derived name with the identical unbounded overflow.

**Design-inherited — surface, don't unilaterally patch** (consistent with how B1/B2 were handled). The crafter faithfully implemented the pinned `ovd-ns-<alloc>` shape and the pinned "≤255" framing; the gap is in D-TME-12. **Recommended resolutions (architect picks one):** (a) bound the netns name too — embed the slot (`ovd-ns-<4hex>`, fully collision-free and traceable via the allocator's slot↔alloc map) or a slot-prefixed truncation of the alloc id; (b) make the derivation fallible with a typed `NetnsNameTooLong` variant (reject at derive, not at the kernel); or (c) accept the 253-char exposure with a real GH issue + a length proptest pinning `plan.netns.len() <= 255` over the full `1..=253` alloc-id-length space, and fix the false "≤255" claim to state the actual bound (`7 + max_alloc_len`).
