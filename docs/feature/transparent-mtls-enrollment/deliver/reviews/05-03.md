## Summary

The RETURN half of this test is honest and the surrounding scaffolding is strong. The connection is driven end-to-end through the **production** composition root — `MtlsInterceptWorker::start_alloc` → the spawned outbound `accept_loop` (getsockname → `resolve(Mesh)` → the real `HostMtlsEnforcement::enforce`), `:1106-1114`) — with only `resolve` injected as a `ScriptedResolve`. The `nft`-dump assertion (`:1118-1122`) proves the *worker*, not the fixture, installed the `iifname VETH_H` egress rule. The encryption oracle (directional 0x17 counts + zero cleartext markers on the leg-B wire, `:1198-1213`) is genuine. The round-trip precondition (`:1174-1192`: byte-exact request received by the mesh peer AND byte-exact response read back by the netns workload) is a real gate and is itself a meaningful backstop. The Q4 authn-only boundary is honoured cleanly (`expected_svid: None` on the resolved arm `:1096`; no intended-peer protection assertion anywhere). Tier-3 hygiene is exemplary and matches the 05-01/05-02 standard (cross-process `flock`, scrub-at-start-and-end, `uname -r` recorded, `strace` required-not-skipped, no `--no-run`). The production substrate is genuinely unchanged (criterion 6 ✓), and the test is structurally a mirror of the surviving inbound sibling.

The blocking problem is confined to the **FORWARD half** — which is the step's *headline* deliverable per criterion 1 ("the forward direction … uses a `write_all` COPY"). The forward-copy oracle, `request_forwarded_through_io_copy` (`:1239-1247`, parser at `:1406-1416`), asserts the request **plaintext marker appears in a traced `write`/`sendto` buffer into a non-splice-source fd**. But the test attaches `strace -f` (`:1141`, `attach_self` runs `strace -f -p <self>`, `:1289-1297`), and the netns workload client is a **forked descendant process** (`run_netns_client` → `Command::new("ip").args(["netns","exec",…,"python3",…])`, `:1018-1024`, spawned at `:1149`, *after* the strace attach at `:1141`). That client sends `OUTBOUND_REQUEST` — **in plaintext** — via `s.sendall(...)` (`:1006`). Under `strace -f`, a forked descendant's `sendto` buffer is captured with the plaintext intact, on a distinct pid. The parser **strips the pid** (`strip_strace_pid_prefix`, `:1439-1443`) and matches the marker against *every* line uniformly. So the netns client's own plaintext `sendall` sets `request_forwarded_through_io_copy = true` **independent of whether the agent's forward pump copied or spliced**. The forward-half oracle therefore does not prove what criterion 1 requires it to prove.

I did not take this on reasoning alone. I ran the exact mechanism under Lima (kernel 7.0):

```

CONFOUND CONFIRMED: forked child's plaintext sendto IS captured under -f

And the `-p`-attach variant the actual test uses (fork happens *after* strace attaches):

```
x4f\x56\x45\x52\x44\x52\x49\x56\x45\x5f\x4c\x41\x54\x45\x46\x4f\x52\x4b   # marker from a post-attach forked child
CONFIRMED: -f -p follows post-attach fork; plaintext captured
# distinct pids in log: 9508 (parent)  9512 (child)

The author clearly *knew* this hazard: the SURVIVING inbound sibling deliberately traces **only** `["recvfrom","splice","read"]` and documents (`mtls_inbound_enforce.rs:253-261`) that "the response S→C copy uses `sendto`/`write`, which are deliberately NOT traced … the response-copy mechanism is the OUTBOUND test's concern, not this one." The inbound test proves its primary direction by *splice presence + the absence of a plaintext read off the ciphertext leg* — a confound-free shape. The outbound test took on the harder presence-of-plaintext-in-a-write proof and then failed to isolate the agent's write from the workload's write. That is the gap.

This matters specifically because of this feature's false-PASS history and because the commit message asserts "Falsifiability confirmed by inverting the forward-copy assertion (REDs)." That falsification is insufficient: inverting `assert!(flag)` to `assert!(!flag)` only proves the flag is *set*; it does not prove the flag is set *by the agent*. The flag is set by the netns client regardless. The claimed falsification cannot distinguish the two producers.

## Findings (priority-ordered)

### `issue (blocking):` S1 — the FORWARD-copy oracle is confounded by the netns workload's own plaintext send; it does not isolate the agent's forward pump

**Location**: assertion `:1239-1247`; parser `request_forwarded_through_io_copy` logic `:1406-1416`; pid-stripping `:1439-1443`; `strace -f` attach `:1141` + `:1289-1297`; netns client plaintext `sendall` `:1006` + `:1018-1024` spawned `:1149`.

**Why blocking**: criterion 1 requires the test to *prove*, via an observable side effect, that the forward direction "uses a `write_all` COPY." The dedicated oracle for that claim can be satisfied entirely by a process that is **not the agent**. `strace -f` follows the forked `ip netns exec … python3` descendant; that descendant's `s.sendall(OUTBOUND_REQUEST)` lowers to a `sendto(fd, <plaintext incl. marker>, …)`; the parser strips the pid and matches the marker on a fd that is never a splice source → `request_forwarded_through_io_copy = true`. **Empirically confirmed under Lima** (both the launch and `-f -p`-post-attach forms — see Summary). Consequently:
- The assertion would pass even if the agent's forward pump used a splice (the inbound shape) instead of a copy — the workload's plaintext `sendall` alone satisfies it.
- The commit's "falsifiability confirmed by inverting the forward-copy assertion" is overstated: inversion proves the flag is set, not that the agent set it.

**Fair mitigation (state it, don't let it excuse the issue)**: a forward-splice-*into-kTLS-TX* regression WOULD be caught by the round-trip assertions (`:1181-1192`) — splicing into a kTLS-TX socket loses records, so the mesh peer would not receive the request byte-exact and `mesh_request_ok`/`client_read` would go RED. So the test as a *whole* is not blind to that one specific regression. But (a) the criterion demands the COPY strategy be proven *as an observable side effect*, and the oracle built for that is unsound; and (b) any forward-copy regression that still delivers the bytes via a *different untraced* mechanism would leave the round-trip green AND the forward-copy oracle green-for-the-wrong-reason. The headline forward-half proof is not load-bearing.

**Required fix (pick one)**:
1. **Filter marker-writes to the test's own thread group.** Retain the pid in the parser and count a marker-carrying `write`/`sendto` only when its TID belongs to the test process's thread group (`Tgid == std::process::id()`, read via `/proc/<tid>/status`). The agent's pump threads are `CLONE_THREAD` threads of the test process (same tgid); the netns `python3` is a separate tgid. This cleanly excludes the workload while keeping every agent pump thread. Then **re-run under Lima and re-confirm falsification by a regression that holds the netns client's plaintext send constant** (the inversion done so far does not).
2. **Adopt the inbound sibling's confound-free shape.** Prove the forward direction by the agent's `read(legF)` of the plaintext request + the *absence* of a splice on the forward path, paired with the leg-B 0x17 encryption already asserted — i.e. assert the mechanism without keying on "plaintext appears in a write buffer," which is the line that admits the descendant's send.

Either way, the docstring claims at `:41-54`, `:160-166`, `:1231-1247`, `:1342-1361` must be brought into line with what is actually isolated.

### `suggestion (non-blocking):` S2 — the `!splice_src_fds.contains(&fd)` guard's stated rationale is largely a red herring, and "NON-splice-source fd" is misleading for a single-fd kTLS leg

**Location**: `:1374-1382`, `:1408-1416`; cross-checked against `mtls/outbound.rs:111` (`arm_ktls_tx_rx(leg_b_fd, …)` — TX **and** RX armed on the **same** fd).

But the mesh peer (`spawn_mesh_peer`, an in-process thread, `:820-826`) writes **ciphertext** (rustls has already encrypted), so it never carries the plaintext marker — the guard does nothing against it. That ordering does hold causally (the request is forwarded before the response can return), but the docstring's framing — "a `write` into a NON-splice-source fd … the forward dst is never a splice source" — is factually wrong for the single-fd kTLS leg and obscures the real (order) dependency. Recommend rewriting the rationale to state the actual invariant (parse-order), and note that the guard does not address the S1 confound at all.

## Findings (priority-ordered)

### `issue (blocking):` F1 — the S1 `TidSampler` attribution races with the forward-pump thread's lifetime; the test is flaky (~29% fail)

**Location**: `TidSampler` (`:1455-1493`, 15 ms `/proc/self/task` poll loop); the live parse `TraceFindings::parse(&raw, test_thread_group)` (`:1442`); the agent-attribution branch (`:1632-1640`); the dependent assertions `:1271` (S3 pin) and `:1289` (forward copy).

**Root cause (executed, not reasoned).** The agent's forward-copy write is performed by the production encrypt pump — `PumpHandle::spawn_encrypt` → `std::thread::spawn` → `run_encrypt_pump`'s `write_all(prelude)` / steady `read(legF)→write_all(legB)` (verified in `crates/overdrive-dataplane/src/mtls/splice.rs:161-172,386-433`). That is an in-tgid thread, so the *design* of the filter is correct. **But the pump thread is short-lived**: once the connection is established the round-trip completes in well under the sampler's 15 ms interval (the mesh peer writes the reply *before* its 300 ms sleep, so the netns client reads its 86 bytes and closes leg F almost immediately, and both pumps then hit EOF and exit). When the pump's entire lifetime falls between two 15 ms samples, the sampler never observes its TID. The `TidSampler` docstring claims it solves exactly this — *"so even a short-lived pump thread that has already exited by `detach_and_read` is captured"* (`:1448-1450`, `:1466-1467`) — but the executed evidence falsifies that claim: a thread shorter-lived than the poll interval is missed ~29% of the time. A 15 ms poll cannot reliably catch a sub-15 ms thread; this is a sampling-vs-lifetime race, not a tuning issue.

**Why blocking.** A Tier-3 acceptance gate that fails ~1 run in 3 on its *own* recorded kernel is not a gate — it is a coin flip. Per `.claude/rules/testing.md`: *"A test that sometimes fails independent of the code under test … Fix flaky tests before adding them"* / *"Flaky … is a bug, not 'just rerun it.' Fix or file."* The `COMMIT EXECUTED PASS` and the commit's quoted `agent_marker_writes=1` are one lucky draw; they do not establish that criterion 1 is provably enforced. CI on the 6.18 matrix inherits the same race (it is userspace timing, not kernel-version dependent) and will redden intermittently.

**Required fix — derive the thread group race-free, not by live sampling.** Sampling `/proc/self/task` is structurally unable to capture a thread shorter-lived than the poll interval. Two robust options, in preference order:

1. **Partition the TIDs from the strace itself.** Add `clone`/`clone3` (and `execve`) to the trace filter. Every thread the test process creates appears as a `clone(… CLONE_THREAD …) = <child_tid>` line emitted *by* a TID already in the test's thread group — so the in-tgid set is the transitive closure of `CLONE_THREAD` clones rooted at `std::process::id()`, computed from the log. The netns client appears as a `clone`/fork **without** `CLONE_THREAD` (+ `execve` of `ip`/`python3`) rooting a separate process subtree — deterministically excluded. The clone events are *permanently in the log* regardless of how briefly the pump lived, so the race is gone. This is the confound-free analogue of what the sampler tried to approximate.
2. **Adopt the inbound sibling's confound-free shape** (the prior review's option 2, still on the table): prove the forward direction by the *absence of a splice on the forward path* + the agent's `read(legF)` of plaintext, paired with the leg-B 0x17 encryption already asserted — i.e. stop keying on "plaintext appears in a write buffer," which is the line that forced the fragile TID attribution in the first place. `mtls_inbound_enforce.rs:253-261` is the precedent; it has never needed thread-group sampling.

Either way: **re-run under Lima ≥10× and require 10/10 green** before re-logging `COMMIT EXECUTED PASS` (a single green run cannot distinguish a fixed oracle from a lucky draw, per the evidence above).

### `suggestion (non-blocking):` F4 — the in-test "falsification" is near-tautological and never executes on the failing path

**Location**: `:1320-1331` (re-parse against the empty thread group).

The empty-tg re-parse asserts `agent_excluded.agent_marker_writes == 0` — but with an empty set, *every* marker write is excluded by construction, so this is true regardless of whether the agent wrote anything. It demonstrates "the filter excludes when the TID isn't in the set," not "the live flag was set by the agent." The genuine criterion-1 proof rests on the *live* `agent_marker_writes ≥ 1` (the racy one) combined with the structural fact that the client's TID is a separate process — which the empty-tg parse does not exercise. Note also that on the ~29% failing path the test panics at `:1271`/`:1289` and **never reaches** the falsification block, so it provides no signal exactly when signal is most needed. A stronger, race-free falsification (once F1 is fixed via clone-tree partitioning): assert the live `excluded_marker_writes ≥ 1` (the client's send is present-and-excluded) AND re-parse against `{the client's recovered pid}` to show the count attributable to the client is excluded — holding the client constant and varying attribution, which is what the commit message claims this does.

## Recommendation

The production substrate, composition, encryption oracle, round-trip, and authn-only handling are sound and must be preserved as-is; this is entirely a test-observability defect.
