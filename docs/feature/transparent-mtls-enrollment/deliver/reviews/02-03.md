## Tier-3 non-vacuity — the load-bearing check this feature kept failing to write

The single most important adversarial question for this feature: **could the Tier-3 assertion pass even if `WriteResolvConf` never ran?** It cannot, and the reason is structural, not incidental:

- `workload_netns_provision.rs:441-449` reads the host `/etc/resolv.conf` *before* provisioning and asserts the responder line (`nameserver 10.99.255.1`, a non-routable test address per `:57`) is **absent** — explicitly so the host-unaffected assertion cannot pass vacuously (the exact lesson from 02-02).
- `:472-477` reads the injected line through `ip netns exec <ns> cat /etc/resolv.conf` — i.e. through the **actual per-netns bind-mount the workload sees**, not the host path the executor wrote. If `WriteResolvConf` were skipped, the netns would have no per-netns file, the bind-mount would not apply, `cat` would fall through to the host file (proven by the precondition to NOT carry the line), and the assertion would FAIL.

This is a genuine load-bearing assertion that kills a "step never ran" mutation. It is the correction of the vacuous-assertion pattern, not a repeat of it.

## Findings

### issue (blocking): false load-bearing premise — three doc/comment sites claim `/etc/netns/<ns>/` "drops with the namespace", contradicting the file's own teardown

Three sites justify the `WriteResolvConf` converge gate with a factually wrong claim about the per-netns dir's lifecycle:

- Converge gate `src/veth_provisioner.rs:787-792`: *"A fresh netns (CreateNetns) **drops the per-netns dir with the old namespace**, so it always re-needs the write."*
- Field doc `src/veth_provisioner.rs:614-616`: *"A fresh netns (`!netns_present`) always re-needs the write — **the per-netns dir is recreated with the netns**."*
- Converge-summary doc `src/veth_provisioner.rs:708-710`: *"… (or the netns was freshly created — **the per-netns dir drops with the old namespace** …)."*

The file's own teardown rationale states the **opposite**, and is correct:

- `src/veth_provisioner.rs:1453-1456` / `:1473-1476`: *"The per-netns resolv.conf dir (`/etc/netns/<netns>/`) is **NOT reaped by `ip netns del`**, so it is removed explicitly — otherwise a re-provision under the same slot would adopt a stale responder line."*

`/etc/netns/<ns>/` is a host-side directory under `/etc`; it has **no lifecycle coupling to the kernel netns object**. The production teardown calls `resolv_conf_dir_remove` (`:1477`) *precisely because* `ip netns del` leaves it behind. The converge-gate justification is the inverse of the truth and contradicts a sibling comment 660 lines away in the same file. The commit message itself even states the correct version ("Teardown reaps `/etc/netns/<ns>/` … `ip netns del` does not").

**Why blocking, not a nitpick** (`.claude/rules/debugging.md` § "Falsifications/premises must be true"; `.claude/rules/verification.md` "unanchored/false claims"; honest-docs discipline): a future maintainer reading `:788` believes the `!observed.netns_present ||` disjunct exists *because the dir vanishes with the netns*. It does not. Acting on that false belief — e.g. concluding the explicit `resolv_conf_dir_remove` teardown step is redundant and deleting it — reintroduces the exact stale-responder-line bug the teardown was written to prevent (a slot reused after a crash would adopt the prior allocation's `nameserver`). In a feature with a documented pattern of rationalised-away regressions, a false premise on the exact gate disjunct under scrutiny is not something to wave through.

**Required fix (docs/comment only; no behaviour change):** correct all three sites to state `/etc/netns/<ns>/` is host-side and reaped *only* by the explicit teardown step (cross-reference `:1473`), and re-justify the `!netns_present ||` disjunct on its true grounds: the observer (`:1545`, `resolv_conf_injected = netns_present && …`) already forces `resolv_conf_injected == false` whenever `!netns_present`, so the disjunct is defence-in-depth mirroring `SetLoopbackUp` (`:780`) — NOT a consequence of the dir's lifecycle.

### Non-blocking

- **issue (non-blocking): `ResolvConfWriteFailed` reused for the teardown removal path.** `resolv_conf_dir_remove` (`:1886`) surfaces a non-`NotFound` removal failure as `ResolvConfWriteFailed`, whose `#[error]` Display (`:197`) says *"writing per-netns resolv.conf … failed"* — wrong verb for a removal during teardown.
