## Findings (priority-ordered, Conventional Comments)

### issue (blocking, security): F2 — concurrent `refresh_index` violates the "single held subscription" invariant; churn is definite, a narrow steady-state row-loss → cleartext window is real

**Location:** `mtls_resolve_adapter.rs:232-268` (`refresh_index`), struct rustdoc 184-196.

The struct rustdoc claims a "**SINGLE long-lived subscription, opened once and HELD for the adapter's lifetime**." The implementation does not uphold this under concurrency — and the 01-01 contract *explicitly sanctions* concurrency: the trait is `Send + Sync + 'static … shared across the worker's per-connection tasks` (mtls_resolve.rs:151-152), and **production runs a multi-threaded runtime** (`overdrive-cli/src/main.rs:35`, `Builder::new_multi_thread()`), so genuinely-parallel `resolve` is the production reality, not a hypothetical.

```rust
let taken = self.subscription.lock().await.take();          // (A) leaves None behind
let mut subscription = match taken {
    Some(existing) => existing,
    None => self.store.subscribe_all().await.map_err(...)?,  // (B) AWAIT point; mutex NOT held
};
// ... drain via now_or_never (no .await) ...
*self.subscription.lock().await = Some(subscription);        // (C) restore — last writer wins
```

Two windows break the invariant:
1. **Initial `None` window (definite).** Two concurrent first-callers both see `None` at (A), both open a fresh `subscribe_all()` at (B), and the second restore at (C) silently drops the first. The `tokio::sync::Mutex` serializes each individual `lock().await` but does nothing for the gap *between* two acquisitions.
2. **Steady-state churn (definite).** The take-then-restore means any two concurrent refreshers contend: one takes (leaving `None`), the other finds `None` and opens a *second* subscription; one is dropped on restore. The "opened once and held" rustdoc invariant is **factually false**.

**Consequence (radical candor on severity):**
- Subscription churn / dropped subscriptions — **definite**. The documented invariant is false as written.
- Row-loss → stale index → cleartext — **real but narrow**. A `service_backends` write delivered only to an about-to-be-dropped subscription (that the surviving later-opened subscription never received) is lost from the index → a should-be-mesh peer at that addr classifies `NonMesh` → **cleartext** (the exact footgun this feature exists to remove). Being honest: *deterministically* constructing a loss is hard — both subscriptions read the same `fan_out` broadcast and `apply_row` is idempotent full-row replacement, so most interleavings are benign. But "hard to force" is not "impossible," and a non-zero silent-cleartext probability on a security boundary is not acceptable on "it's unlikely."

**Why the suite misses it:** all 7 `#[tokio::test]` use the default current-thread runtime, and the PBT uses `new_current_thread()` (546). A single-threaded executor never interleaves two `resolve` futures across the (B) await — the bug is structurally invisible to the tests.

**Not covered by #237.** #237 is scoped to the boot/restart *cold-start* window (index empty until `BackendDiscoveryBridge` re-converges; mitigated by 04-02 subscribe-before-first-write). F2 is a **steady-state concurrency** defect on the warm path and must not hide behind #237.

**Fix:** open the subscription **exactly once** (in `new`/`probe`) and make the `None → Some` transition atomic / single-winner so concurrent refreshers cannot each re-subscribe or `take`-and-replace the durable subscription (`.claude/rules/development.md` § "Check-and-act must be atomic"). At minimum, if v1 chooses to accept the risk, the rustdoc must stop asserting "opened once and held" (it is false today), and the churn + steady-state cleartext window must be surfaced to the user and tracked in an issue **distinct from #237**.

### question (non-blocking, traceability): F5 — `healthy`-vs-`running`, and the "STOP-and-surface" instruction

The step name + criterion 1 say "reads `service_backends` ∩ **running**"; implementation_notes said "confirm the running-intersection source (alloc_status running set); **if ambiguous, STOP and surface the gap** (do not invent a running-set reader)." The impl classifies on `Backend.healthy` (160-168), not an alloc_status join. **Disposition: DEFENSIBLE, confirmed against code** — `BackendDiscoveryBridge` builds backends from `actual.running` only (`backend_discovery_bridge.rs:343-353`) and `service_lifecycle` skips non-`Running` allocs (`service_lifecycle.rs:827-829`), so every `Backend` in a row is already a running alloc and `healthy` is the readiness refinement mapping to `MeshUnreachable`.

Two caveats worth surfacing:
1. The other production writer, `BackendDiscoveryBridge` (Phase 2.2 single-node), **hardcodes `healthy: true`** (`// GH #170 ships real health`, line 351). Through that writer, `MeshUnreachable` is effectively unreachable today (nothing emits `healthy:false`). The adapter's arm is correct and future-proof, but its production reachability is writer-dependent — worth a one-line note.
2. The roadmap's explicit "STOP and surface if ambiguous" gate appears resolved by self-investigation (memory #44966) rather than a recorded wave-decision. The resolution is correct, but a silent self-resolution of an explicit STOP gate is a process smell — confirm it was surfaced.

### suggestion (non-blocking, security-adjacent): F4 — steady-state broadcast lag-drop is a real stale-index source, NOT in #237

The single held `subscribe_all` is the **whole observation firehose**, one bounded broadcast channel (capacity 1024), drained only on probe/resolve via `now_or_never`; a `Lagged` error is **silently filtered** (`ok_or_skip` / `filter_map(Result::ok)`; both adapters). If 1024 observation rows of any kind are written between two drains of a quiet adapter, a `service_backends` update can be evicted from the ring → permanently missing from the index → stale → `NonMesh` → cleartext. Distinct from #237's boot window and from F2's churn — a capacity/drain-cadence hazard on the warm path. Surface to the user; if accepted for v1, track in the same hardening bucket as #237 (separate line item), and consider a rustdoc note that the held subscription assumes drain cadence keeps pace with the firehose.
