## Findings (priority-ordered, Conventional Comments)

### issue (blocking, security): F2 ‚Äî concurrent `refresh_index` violates the "single held subscription" invariant; churn is definite, a narrow steady-state row-loss ‚Üí cleartext window is real

**Location:** `mtls_resolve_adapter.rs:232-268` (`refresh_index`), struct rustdoc 184-196.

The struct rustdoc claims a "**SINGLE long-lived subscription, opened once and HELD for the adapter's lifetime**." The implementation does not uphold this under concurrency ‚Äî and the 01-01 contract *explicitly sanctions* concurrency: the trait is `Send + Sync + 'static ‚Ä¶ shared across the worker's per-connection tasks` (mtls_resolve.rs:151-152), and **production runs a multi-threaded runtime** (`overdrive-cli/src/main.rs:35`, `Builder::new_multi_thread()`), so genuinely-parallel `resolve` is the production reality, not a hypothetical.

```rust
let taken = self.subscription.lock().await.take();          // (A) leaves None behind
let mut subscription = match taken {
    Some(existing) => existing,
    None => self.store.subscribe_all().await.map_err(...)?,  // (B) AWAIT point; mutex NOT held
};
// ... drain via now_or_never (no .await) ...
*self.subscription.lock().await = Some(subscription);        // (C) restore ‚Äî last writer wins
```

Two windows break the invariant:
1. **Initial `None` window (definite).** Two concurrent first-callers both see `None` at (A), both open a fresh `subscribe_all()` at (B), and the second restore at (C) silently drops the first. The `tokio::sync::Mutex` serializes each individual `lock().await` but does nothing for the gap *between* two acquisitions.
2. **Steady-state churn (definite).** The take-then-restore means any two concurrent refreshers contend: one takes (leaving `None`), the other finds `None` and opens a *second* subscription; one is dropped on restore. The "opened once and held" rustdoc invariant is **factually false**.

**Consequence (radical candor on severity):**
- Subscription churn / dropped subscriptions ‚Äî **definite**. The documented invariant is false as written.
- Row-loss ‚Üí stale index ‚Üí cleartext ‚Äî **real but narrow**. A `service_backends` write delivered only to an about-to-be-dropped subscription (that the surviving later-opened subscription never received) is lost from the index ‚Üí a should-be-mesh peer at that addr classifies `NonMesh` ‚Üí **cleartext** (the exact footgun this feature exists to remove). Being honest: *deterministically* constructing a loss is hard ‚Äî both subscriptions read the same `fan_out` broadcast and `apply_row` is idempotent full-row replacement, so most interleavings are benign. But "hard to force" is not "impossible," and a non-zero silent-cleartext probability on a security boundary is not acceptable on "it's unlikely."

**Why the suite misses it:** all 7 `#[tokio::test]` use the default current-thread runtime, and the PBT uses `new_current_thread()` (546). A single-threaded executor never interleaves two `resolve` futures across the (B) await ‚Äî the bug is structurally invisible to the tests.

**Not covered by #237.** #237 is scoped to the boot/restart *cold-start* window (index empty until `BackendDiscoveryBridge` re-converges; mitigated by 04-02 subscribe-before-first-write). F2 is a **steady-state concurrency** defect on the warm path and must not hide behind #237.

**Fix:** open the subscription **exactly once** (in `new`/`probe`) and make the `None ‚Üí Some` transition atomic / single-winner so concurrent refreshers cannot each re-subscribe or `take`-and-replace the durable subscription (`.claude/rules/development.md` ¬ß "Check-and-act must be atomic"). At minimum, if v1 chooses to accept the risk, the rustdoc must stop asserting "opened once and held" (it is false today), and the churn + steady-state cleartext window must be surfaced to the user and tracked in an issue **distinct from #237**.

## Findings (priority-ordered, Conventional Comments)

### issue (blocking, security): F-A ‚Äî `BackendIndex` is keyed by `addr` alone; a backend `addr` shared by two services gets wrongly evicted on the other service's shrink ‚Üí long-lived stale-index ‚Üí cleartext

**Location:** `mtls_resolve_adapter.rs:188-235` (`BackendIndex::apply_row` / `replace_from_snapshot`; the global `by_addr: BTreeMap<SocketAddrV4, Backend>` + per-service `addrs_by_service`).

`apply_row` evicts a service's prior addrs unconditionally from the **global** `by_addr` map:

```rust
if let Some(stale) = self.addrs_by_service.remove(&service_id) {
    for addr in stale { self.by_addr.remove(&addr); }   // unconditional global remove
}
```

If one backend `addr` X is a member of **two** services A and B (both `service_backends` rows carry X ‚Äî `service_lifecycle::readiness_backend_row_action` builds each service's row independently from *its own* `actual.allocs`, `service_lifecycle.rs:825-860`), then:

1. `apply_row(A,[X])`, `apply_row(B,[X])` ‚Üí `by_addr={X:‚Ä¶}` (last writer wins), `addrs_by_service={A:[X],B:[X]}`.
2. A's set shrinks: `apply_row(A,[])` ‚Üí evicts A's stale `[X]` ‚Üí **`by_addr.remove(X)`** ‚Äî even though B still owns X.
3. `resolve(X)` ‚Üí miss ‚Üí **`NonMesh` ‚Üí cleartext**, for a workload that is still a healthy mesh backend via B.

This is **not** self-healing on the warm path: the index only rebuilds on a relist (probe / watch-close / `Lagged`). With no lag episode and no B-side change, the wrong eviction persists **indefinitely** ‚Üí an unbounded silent-cleartext window on the exact boundary this feature exists to remove. A second symptom of the same root cause: when A and B disagree on `X.healthy`, `by_addr`'s last-writer-wins makes X's classification depend on watch interleaving order (a coherence/determinism smell).

**Honesty on reachability (applying the same bar the prior F2 used).** The defect in the index logic is demonstrable in isolation; its *production* reachability hinges on whether one alloc/addr can be a backend of ‚â•2 services in v1 headless. I could **not** confirm this is reachable, and I could **not** confirm it is forbidden ‚Äî the design (D-TME-11 / C4) pins an `addr ‚Üí Backend` reverse index but **never states an "addr belongs to at most one service" precondition**, and the `BackendIndex` rustdoc (179-199) does not name it. "Hard to reach" is not "impossible," and a non-zero, potentially-unbounded cleartext path on a security boundary is not acceptable on "it's unlikely" ‚Äî the standard the prior review correctly applied to F2.

**Fix (cheapest first):** (a) confirm one-addr-‚Üî-one-service is a v1 invariant of the writers, **pin it as a documented precondition on `BackendIndex` + the trait, and add a test** asserting the index's defined behavior under a shared addr; OR (b) if it is not invariant, fix the structure so eviction is ownership-aware (refcount addrs by contributing service, or only `by_addr.remove(addr)` when no other service still claims it). Either way this must be resolved or explicitly surfaced to the user before PR.
