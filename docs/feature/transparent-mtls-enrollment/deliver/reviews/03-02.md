## issue (blocking): step 03-02 left two categorically-false statements in production code (`mtls_intercept_worker.rs`) with no in-tree marker

Changing `accept_outbound_leg` to recover `peer` via `getsockname` (and ignore
the declared `_peer` arg) **inverted the truth of documentation in the caller**,
`crates/overdrive-worker/src/mtls_intercept_worker.rs` — a file 03-02 did NOT
touch (correctly: it is 04-02's `files_to_modify`). Two sites now assert the
opposite of what the tree does:

1. **Field doc, `real_peer` (mtls_intercept_worker.rs:~203-208):**
   > "The kernel rewrite is lossy (unlike inbound TPROXY, **the original
   > destination is NOT recoverable via `getsockname` on the accepted leg-F
   > socket**), so the worker cannot observe `real_peer` from the connection
   > alone."

   This is now categorically false. 03-02 — and design D-TME-4 — establish that
   under the Path A nft-TPROXY mechanism the outbound orig-dst **is** recoverable
   via `getsockname`, exactly as inbound. The sentence describes the *retired*
   `cgroup_connect4`-rewrite lossiness (D-TME-3 RETIRED), not the shipped
   mechanism.

2. **Call-site comment (mtls_intercept_worker.rs:657-660):**
   > "built into `Routed::Outbound { peer: real_peer }` so `enforce` dials the
   > REAL peer."

   Also false: `accept_outbound_leg` now builds `Routed::Outbound { peer }` from
   the **getsockname-recovered** addr, not from `real_peer` (the passed `peer` is
   discarded as `_peer`).

**Why this is blocking, not cosmetic.** This is the precise defect class this
feature keeps shipping — the 03-01 adversarial review caught "false design
citations," and the project rule is unambiguous: *"No aspirational docs. Never
document behaviour that is not implemented."* A reader of `mtls_intercept_worker.rs:203`
during the 03-02→04-02 window reads a confident, wrong claim about the core
recoverability premise of the whole feature, with **no breadcrumb** that it is
transitional. And 04-02's timing is not guaranteed safe to lean on: this
feature's execution log already shows sequencing turbulence (02-05 SUPERSEDED;
a `BLOCKED_BY_DEPENDENCY` on 02-05) — "04-02 will delete it soon" is exactly the
deferral-without-a-marker the repo distrusts.

**Scope tension — acknowledged.** The crafter correctly stayed inside 03-02's
file scope; the declared-peer *surface* (param, slot, and these docs) is owned
by 04-02 ("DELETE the declared-peer surface in the SAME commit"). I am **not**
asking for the 04-02 deletion here. The commit message does breadcrumb the
*slot* removal — but not the *doc-correctness* inversion, and a reader of the
source in isolation never sees the commit message.

**Minimal, scope-respecting remedy (pick one — both are cheaper than a rework
cycle, neither is the 04-02 deletion):**
- (a) Add a one-line transitional marker at each site, e.g.
  `// TRANSITIONAL (03-02): outbound orig-dst IS now getsockname-recovered in accept_outbound_leg; this declared-peer doc/slot is deleted in 04-02 (#…).`
  A comment correction is within the blast radius of the step that caused the
  staleness and does not delete the deferred surface. — OR —
- (b) If touching the 04-02-owned file is off-limits, record in the execution

## suggestion (non-blocking): the root-gated AT is a near-duplicate of the unit test for this step

`worker_intercept_install_leg_acquire_outbound` and the new default-lane unit
test run the **same** plain-loopback scenario with the **same** discriminating
power: both prove "recovered peer == dialed addr, != declared arg." On loopback,
`getsockname == listener.local_addr == connected-local-addr == dialed_addr` all
coincide, so neither test can distinguish `getsockname_orig(fd)` from
`leg_f.local_addr()`. The genuinely-discriminating proof (TPROXY makes
`getsockname` ≠ the socket's bound addr) is 03-03's Tier-3 — which is correct
per the step. Action: make sure 03-03's Tier-3 assertion actually pins
`getsockname == dialed-dst` against a dialed-dst that **differs** from the
leg-F bound addr, so the recovery mechanism is discriminated, not just exercised.
