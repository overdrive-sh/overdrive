## Summary

This is a carefully built, genuinely rigorous Tier-3 test: it drives a real
workload `connect()` from inside a netns through real nft-TPROXY, captures the
wire with a real `AF_PACKET`/`SOCK_RAW` socket, counts directional `0x17`
TLS-1.3 application_data records, exercises all three Q3 resolve arms, proves
F5 self-exempt-impossible, runs both a NORMAL and a TRACED timing regime, and —
critically — drives the **real `HostMtlsEnforcement` substrate** (ADR-0069),
not a hand-rolled TLS stack. The execution log is honest (a real
`BLOCKED_BY_DEPENDENCY` was logged, not a false PASS — see praise).

The blocking problem is structural and is the one the step's name exists to
foreclose: **the walking skeleton never calls the production composition root
`start_alloc` / `accept_loop`.** It rebuilds leg-F/leg-C with
`make_transparent_listener` in the test, installs the rules via the seam
functions directly, and hand-rolls the per-connection accept→recover→resolve→
enforce loop (`drive_outbound_once`). That substitution is *exactly* the
mechanism that masked the production leg-F plain-bind `ConnectionRefused` bug —
which this step's own GREEN phase surfaced as a "fixture bug," fixed in the
test rather than in production, and which was only repaired in the post-commit
follow-up `3c085e5d`. Given the feature's documented false-PASS / production-
masking history (02-02 vacuous sysctl asserts, 03-03 F5 overclaim, the
leg-F plain-bind gap itself), blessing this pattern is the wrong call.

## Issues

### `issue (blocking):` C1 — the walking skeleton substitutes the production composition root instead of composing it

**The contract.** Roadmap step `05-01`, criterion 1 + `implementation_notes`:
the deliverable composes "02 … 03 … **04 (start_alloc both rules + resolve
consumer + C3 netns wiring)**" into a full bidirectional mTLS round-trip on the
Path-A mechanism. Criterion 7 permits the worker-crate home ("or
overdrive-control-plane if full boot composition is needed") — but that clause
is about *where the test lives / which harness it re-authors*, not a licence to
skip driving `start_alloc`. Project CLAUDE.md "Implement to the design" binds
the deliverable to the named composition.

**What the test actually does** (`bidirectional_walking_skeleton.rs`):
- `:1401` / `:1586` / `:1752` — builds leg-F and leg-C with
  `make_transparent_listener(...)` **in the test**.
- `:1410` / `:1596` — installs the egress/ingress rules by calling
  `install_outbound_tproxy` / `install_inbound_tproxy` **directly**.
- `:1815-1860` `drive_outbound_once` — **hand-rolls** the per-connection loop:
  `accept_outbound_and_recover_orig_dst` (`:1825`) → `resolve.resolve` (`:1833`)
  → the three-arm `match` → `adapter.enforce` (`:1844`).
- `:828-847` `ScriptedResolve` — a test double substituted for the resolve port.

**What production does** (`mtls_intercept_worker.rs`):
- `start_alloc` (`:339`) binds leg-F via `make_transparent_listener` (`:368`,
  post-`3c085e5d`), installs the egress rule (`:400`), and spawns
  `accept_loop` via `spawn_accept_loop` (`:542`/`:570`).
- `accept_loop` is the **real composition root**: it owns the blocking
  accept → `accept_outbound_and_recover_orig_dst` → resolve → the three-arm
  branch (`Mesh`→`spawn_enforce` `:701`; `NonMesh`→`spawn_cleartext_passthrough`
  `:706`; `MeshUnreachable`→drop-fail-closed `:709-717`).

So `drive_outbound_once` is a **replica** of `accept_loop`, not a driver of it.
The walking skeleton proves the *seam functions* and the *enforce substrate*
compose; it does **not** prove that `start_alloc` / `accept_loop` — the code
that ships — composes them.

**Why this is blocking, not cosmetic.** This is the precise substitution that
masked a real production bug. Pre-`3c085e5d`, `start_alloc` bound leg-F with a
plain `TcpListener::bind("127.0.0.1:0")`; under the non-rewriting egress
`tproxy to 127.0.0.1:<legF>` divert the kernel delivers orig-dst-addressed
packets a plain socket cannot receive → `ConnectionRefused`. **No
connection-driving test caught it** because every one (this skeleton included)
builds a *correct* transparent leg-F in-test. The step's own GREEN phase even
recorded "transparent leg-F is mandatory for the TPROXY redirect" as a *fixture
bug* (commit body) — i.e. it diagnosed the production constraint and fixed the
test rather than production. The bug shipped in `cb7d8d09` and was repaired only
in the post-commit `3c085e5d`.

**Does `3c085e5d` close the gap? Partially — not enough.** Its guard
`start_alloc_legf_must_be_ip_transparent_for_real_tproxy_traffic`
(`start_alloc_installs_both_tproxy.rs`) genuinely drives production
`start_alloc` with a real netns connect and a litmus (revert to plain bind →
RED). That is the right shape and is good work. But:
1. It asserts via a **spy `SimMtlsEnforcement`** (records that `enforce` was
   called with `Routed::Outbound { peer == dialed }`) — it does **not** prove a
   full mTLS handshake, a `0x17` wire round-trip, or no-RST through production.
2. It drives only the **Mesh** arm; the production `accept_loop`
   `NonMesh` (cleartext relay) and `MeshUnreachable` (fail-closed drop) arms are
   **never driven end-to-end with real traffic** — only the test's hand-rolled
   copy in `drive_outbound_once` exercises those.
3. The walking skeleton itself is unchanged by `3c085e5d` — it still
   substitutes leg-F and the loop.

Net: **no single test proves production `start_alloc`/`accept_loop` through a
full bidirectional mTLS round-trip + `0x17` capture**, and the production
non-Mesh arms have no end-to-end Tier-3 coverage. A future regression in
`accept_loop`'s wiring (the same class as the leg-F bug) would pass this step
green.

**Resolution (either path closes it):**
- **(preferred)** Point the walking skeleton at `MtlsInterceptWorker::start_alloc`:
  drive a real workload connect through the production bind + production
  `accept_loop`, and keep the existing observable oracles (O1 getsockname, O2
  `0x17`, O3 byte-exact, O4 no-RST). This may need a small testable seam on the
  worker (e.g. an accept-one variant) — design that seam to the existing port
  surface; do **not** invent unrelated public API (CLAUDE.md "never invent API
  surface").
- **(acceptable)** Explicitly accept the **split coverage** as a design
  decision — walking skeleton = substrate + full round-trip with substituted
  plumbing; `3c085e5d` guard = production `start_alloc` wiring with spy enforce —
  and record the residual gap (production `accept_loop` through a full
  round-trip + the `NonMesh`/`MeshUnreachable` arms end-to-end) as a tracked
  GitHub issue cited at both test sites. Per CLAUDE.md the issue requires
  **user approval before creation** — surface it; do not create it unilaterally.

Whichever path: the commit/log language must stop claiming `start_alloc` is
"composed" until it is actually driven (see C2).

### `suggestion (non-blocking):` C2 — `start_alloc` / `MtlsResolve (01-02)` named as "composed" when they are replicated / doubled

The `cb7d8d09` commit body and the roadmap notes name "MtlsResolve (01-02)" and
"start_alloc … resolve consumer" among the composed seams, but the test uses
`ScriptedResolve` (`:828`) and never calls `start_alloc`. The three resolve arms
*are* exercised, so the decision behaviour is proven against a double — but the
production resolve index (01-02/01-03 List-then-Watch) and `start_alloc` are
not. Given this feature's documented overclaim history, tighten the narrative:
say "replicates the `accept_loop` shape against a `ScriptedResolve` double and
the real enforce substrate," not "composes `start_alloc` + `MtlsResolve`." This
is the honesty half of C1; it stops being an issue once C1 is resolved by
actually driving the production path.

### `praise:` / `suggestion (non-blocking):` C3 — confidentiality oracle scoping is sound but the comment oversells it

`scan_frames` (`:677-726`) counts `plaintext_marker_hits` only on streams with
`records > 0` (`:720`), exempting the single-`lo` cleartext leg-S stream
(agent↔S, which legitimately carries the markers because S is a plaintext
workload). This scoping is **correct** for the topology. But note for the record
that, taken alone, `plaintext_marker_hits == 0` is near-vacuous as an
independent encryption oracle — a *fully* cleartext leg-B/leg-C would have
`records == 0` and be exempted. The load-bearing proof is `has_app_data()` +
`records_to_wire_port > 0` **and** `records_from_wire_port > 0` (`:1502-1513`),
which a cleartext leg fails. That combination *does* hold, so O2 is sound.
Suggestion: trim the comment (`:706-719`) so it does not imply the plaintext
counter is the primary confidentiality proof — the directional `0x17` counts are.

### `suggestion (non-blocking):` C4 — O4 "no RST" leans partly on pump-internal `liveness()`

`adapter.liveness(&handle) == PumpLiveness::Running` (`:1457`, `:1653`) is a
read of pump-internal state; `.claude/rules/testing.md` § Tier-3 cautions
against asserting on "program internal reachability." It is adequately mitigated
here by the **byte-exact round-trip** (`:1486-1496`, `:1664-1677`) and the
inbound client's `observed_rst == false` (`:1678`) — both genuinely observable.
Reframe the liveness assertion (in the comment) as *corroborating*, not the
primary no-RST oracle.

### `suggestion (non-blocking):` C5 — inbound F5 is argued, not directly tested

The OUTBOUND self-exempt-impossible is directly exercised
(`prove_workload_cannot_self_exempt`, `:1738` — a SO_MARK-stamped netns dial is
still captured to leg-F). The INBOUND counterpart is only argued by analogy
(`:1712-1723`). The argument (the mark is skb-local; it does not cross the
veth/netns boundary) is correct, and the inbound leg-S non-recapture is
implied by the inbound round-trip completing — but criterion 6 names both
directions. A direct inbound self-exempt test would close the asymmetry. Low
priority.

## New findings (second pass)

### `suggestion (non-blocking):` R1 — `leg_c_addr` is sound, but the "twin of `leg_f_addr`" / "#178 will consume" language overclaims

The accessor itself clears the "test-only public API" bar: it is **architect-
ratified** (D-TME-13, `wave-decisions.md:1512`), the design record *explicitly*
runs the test-only-smell check and the leg-C/leg-F asymmetry argument rather than
rubber-stamping, it exposes **only a `SocketAddrV4`** (no SVID/key/identity —
inside the authn-only v1 boundary), and it is genuinely *necessary* to drive the
production inbound `accept_loop` (the alternative is the seam-replication C1
rejected). The crafter implemented the pinned signature exactly. This is **not**
an invented-API violation — it went through the architect channel with reasoning
that anticipated this exact concern.

What overclaims is the *narrative*, and it is the same overclaim tic this feature
keeps repeating (02-02 vacuous asserts, 03-03 F5, the C2 "composed" language):

1. **"the symmetric twin of `leg_f_addr`"** (field doc `mtls_intercept_worker.rs:219`,
   `:440`; commit body). There is **no public `leg_f_addr`** — leg-F's addr is a
   *local* in `start_alloc` (`:396`) consumed *inline* by
   `install_outbound_tproxy(host_veth, leg_f_addr.port())` (`:418`), never stored
   on `AllocIntercept`, never exposed. The "twin" is the *capture pattern*, not a
   public-accessor sibling. The wave-decision (`:1675-1698`) is honest about this
   asymmetry; the in-code field comment blurs it.
2. **"the SAME read-site #178's production inbound-redirect install will consume"**
   stated as fact (docstring `:858` region; `:1626`; `wave-decisions.md:1663-1667`).
   By the leg-F precedent the architect itself cites, #178's inbound install — if
   it mirrors leg-F and installs in `start_alloc` — would read the **inline local**
   `leg_c_addr` (already present at `:447`), *not* `self.leg_c_addr(alloc)`. Whether
   #178 consumes the public accessor or an inline local depends on #178's
   install-site/timing, which is unresolved (`#178` is the orig-dst→backend
   resolution; its inbound-rule install timing is its own design). So the accessor
   may remain test-observability-only.

Neither undermines correctness. Recommend softening both to the honest framing
the wave-decision already half-states: *"the v1 inbound test-observability seam;
#178 is **expected** to reuse it for the production inbound-redirect install
pending that install's site/timing design."* This keeps the feature's
overclaim-discipline consistent and avoids a future reader treating
`leg_c_addr`'s production-consumer as settled.

### `suggestion (non-blocking):` R2 — the MeshUnreachable arm proves fail-closed by sentinel-silence + cross-arm liveness, not by a positive interception signal

The fail-closed arm (`:1550-1572`) binds a sentinel on `unreachable`, drives a
netns dial, then asserts `fc_listener.accept().is_err()` — "no connection reached
the should-be-mesh peer." This is **load-bearing as a set**: the NonMesh arm
(`:1578-1595`) proves the production relay path *does* reach an upstream when it
should, so MeshUnreachable's silence is a real differential (drop vs relay), and
the Mesh/NonMesh round-trips in the same function prove leg-F/`accept_loop` are
live. So it is not vacuous *in context*.

But `let _ = fc_client.join()` (`:1562`) **discards the workload's own connect
outcome** — the one signal that would distinguish "production accepted leg-F →
resolved MeshUnreachable → dropped fail-closed" from "the connection was never
intercepted at all." The arm's *positive* claim (interception-then-drop) is
inferred from sentinel-silence plus cross-arm liveness, not asserted directly
(debugging.md §8 "let _ on fallible", §11 "confirm the producer ran"). If a
future regression broke MeshUnreachable-specific interception while Mesh/NonMesh
stayed green, this arm could pass on silence. Strengthen with a positive signal:
assert the production enforced-list recorded the leg-F accept-then-drop, or that
the workload observes *accept-then-RST* (intercepted, then fail-closed) rather
than a bare connection failure.

### `nitpick (non-blocking):` R3 — `.ok()` on `set_nonblocking` + single-poll-after-sleep

`fc_listener.set_nonblocking(true).ok()` (`:1556`) swallows a fallible setup
(debugging.md §8). The blast radius is benign here — a failure would make
`accept()` *block* (a hang, not a false green) — but the same `let _`/`.ok()`
smell the rules flag. Relatedly, the fail-closed check is a single `accept()`
poll after a fixed `sleep(600ms)` (`:1565-1566`): an absence-of-evidence shape
(debugging.md §11). Low risk on loopback (the NonMesh arm shows the relay is
near-immediate), but a short poll-loop would be more honest than one fixed sleep.
