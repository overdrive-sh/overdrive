## Blocking

### issue (blocking): `sweep_per_workload_tproxy_rules` swallows *every* `list_chain` failure into `Ok(0)` — the named "distinct-failure-modes" anti-pattern

`crates/overdrive-worker/src/mtls_intercept.rs:457`

```rust
let Ok(dump) = list_chain() else {
    return Ok(0);
};
```

`list_chain()` returns a single undifferentiated `InterceptError::TproxyInstall`
for **every** non-success — chain/table absent (benign: "nothing to sweep"),
*and* a genuine `nft` failure (binary missing, `EPERM`, transient lock/resource
error). This `let Ok(..) else` collapses all of them to `Ok(0)`. That is exactly
the shape `.claude/rules/development.md` § "Distinct failure modes get distinct
error variants. Never silently absorb a `Result` into a default" forbids — its
own worked example is structurally identical (`read_to_string(...).unwrap_or_default()`),
and its rule is: absorb the *benign* `ErrorKind` (here: absent-table, the
`NotFound` analog) into the default, but **propagate** every other kind.

Why this is more than cosmetic — the safety-net argument has a hole *in §5's
core case*. The inline comment (`:454-456`) defends the swallow with: "A genuine
`nft` failure on the SAME boot would also fail the subsequent `start_alloc`
install loudly." But a **surviving Running workload does not trigger a
`start_alloc`** — SPIKE-B (and the recovery design itself) establishes the
reconciler does NOT re-drive Running survivors. So on a survivor-recovery boot,
a genuine transient `nft list` failure → `Ok(0)` → the survivor's guard-less
stale rule is left stranded (the D2 dead-weight §5 exists to reap), with **no
downstream `start_alloc` to catch it**. The safety net covers the one case (fresh
alloc) that doesn't need §5 and misses the one case (survivor) that does.

Secondary: this makes §5's fail-posture internally inconsistent — a by-handle
`nft delete` failure refuses the boot (`:471` propagates → `NftRuleSweep` →
`health.startup.refused`, fail-CLOSED), but a `list` failure proceeds
(fail-OPEN). The design text (`lib.rs:2003`, "A sweep failure … refuses the
boot, same fail-closed posture as the adopt conflict") describes the
fail-closed posture; the list path silently violates it.

**Blast radius is genuinely narrow** (a transient non-absent `nft list` failure
at the exact boot instant, with guard-less survivor rules present) and the
failure degrades to *pre-§5 behaviour* — a stranded dead-weight rule, self-
corrected on the next clean boot; nothing is torn down. So this is the lightest
kind of blocking: a named-rule violation with a real-but-bounded correctness gap,
not a boot-breaker.

**Fix (small, feasible — the stderr is already captured):** distinguish in
`list_chain` (or at the call site) the absent-table shape (`nft` emits `Error:
No such file or directory` / "does not exist" to stderr) → `Ok(0)`, from a
genuine failure → propagate the `TproxyInstall` so the boot refuses, matching the
delete-path posture and the rule's prescribed `match { Ok | Err(absent) => default
| Err(_) => return Err }` shape. Adjust the docstring/inline comment to stop
implying the current code distinguishes the two ("the chain/table absent is
treated as Ok(0), **distinguished by list_chain's success/failure**" — it is not
distinguished; all failures map to Ok(0)).

(If the team consciously *wants* fail-open-on-list, that is a design call — but
then it needs an explicit, sanctioned exception comment citing the rule and
justifying why a survivor's stranded rule on a transient `nft` hiccup is
acceptable, not a docstring that claims a distinction the code does not make.)

### question (non-blocking): does the global-netns-scanning recovery pass risk reaping a *live* netns whose owner isn't in this boot's Running set?

`veth_provisioner.rs:1943-1983` (`adopt_observe`) + `:2024-2031` (orphan GC)

`adopt_observe` enumerates **every** `ovd-ns-*` netns on the host and classifies
any whose inode does not match a PID of a `state == Running` alloc in *this
boot's* `ObservationStore` as an orphan → `teardown_workload_netns` (real `ip
netns del`). Two scenarios worth confirming are safe:

1. **Production timing**: on a fresh CP restart, is `obs.alloc_status_rows()`
   guaranteed converged (all survivor Running rows present) *before* the recovery
   pass runs? If a survivor's row is momentarily absent/non-Running at the
   boot-recovery instant, its live netns is GC'd — tearing down a live workload's
   network. (Single-node durable `LocalStore`/Corrosion likely makes this safe
   since rows persist across restart and are read synchronously — but the
   correctness rests on that ordering, and it isn't asserted anywhere.)
2. **Lima parallelism**: nextest runs Tier-3 tests in parallel within a binary.
   If test A boots `run_server` (mtls on) and runs the recovery pass while test B
   holds a live `ovd-ns-<slot>`, A's obs store has no B alloc → A classifies B's
   netns as an orphan → GCs B's live netns. The suite is reportedly GREEN on Lima
   7.0.0, so this either doesn't overlap at the boot-recovery instant or the
   slot ranges are disjoint in practice — but it's an introduced cross-test
   contamination surface. Please confirm (disjoint slots / serialization / B sets
   `dataplane_override` so the recovery pass is skipped) and, given the user's
   "Lima ground-truth gate run on phase-06 Tier-3 steps" note, a fresh full-suite
   Lima run before landing would close it.

## NEW Blocking

### issue (blocking): the orphan-GC observe path swallows fallible cgroup/proc reads into "no live PID" → tears down a **live** workload's netns — the same anti-pattern the §5 fix just closed, with a *destructive* downstream

`crates/overdrive-control-plane/src/veth_provisioner.rs` — `adopt_observe`
(`:1943-1992`), fed by `alloc_scope_pids` (`:1920-1928`), classified by
`plan_adopt_actions` (`:1819-1827`), executed by `teardown_workload_netns`
(`:1758-1770`).

The orphan-GC decision is "a surviving `ovd-ns-<slot>` whose slot is **not
correlated to a Running alloc** → GC via `ip netns del`." Correlation runs
through three fallible reads, **each of which swallows its error into the same
default that drives the destructive branch**:

1. **`alloc_scope_pids:1924`** — `let Ok(body) = std::fs::read_to_string(&procs)
   else { return Vec::new() };`. A Running alloc whose `cgroup.procs` read fails
   for *any* reason (EACCES, EIO, a transient, or a cgroup-path regression)
   contributes **zero** PIDs — indistinguishable from a genuinely empty scope.
2. **`netns_file_inode:1955`** — `if let Some(ino) = … { by_inode.insert(…) }`.
   A surviving netns whose inode `stat` fails is never entered into `by_inode`,
   so no live PID can ever match it → it stays `owner: None`.
3. **`read_proc_netns_inode:1982`** — `if let Some(ino) = …` silently skips a PID
   whose `/proc/<pid>/ns/net` read fails.

Any of the three turns a *live, Running* workload into an apparent orphan.
`plan_adopt_actions:1822-1824` is a pure `Some => adopt / None => gc` with **no
independent "no live PID" re-check**, and `teardown_workload_netns:1758-1769`
calls `netns_del` (`ip netns del`) **immediately, with no live-PID guard**. So
there is **no downstream defense** — a swallowed read deletes the named netns of
a running workload, returns its slot to the free pool, and lets a future `assign`
reuse that slot (a fresh `ovd-ns-<slot>` colliding with the orphaned-but-live
workload's actual network).

**Why this is blocking, and why parity with the first pass demands it:**

- It is the *exact* rule violation the first pass blocked on — `.claude/rules/
  development.md` § "Distinct failure modes get distinct error variants. Never
  silently absorb a `Result` into a default value" — and `debugging.md` § 8
  (`let _` / swallow on fallible setup). The §5 `list_chain` swallow was blocked;
  this is the *same* swallow in the sibling path.
- The downstream is **strictly worse**. The §5 case stranded a dead-weight nft
  rule (self-corrected on the next clean boot, nothing torn down). This case runs
  a **destructive** `ip netns del` against a live workload — the asymmetry (fix
  the dead-weight-rule swallow, leave the tear-down-live-netns swallow) is the
  decisive argument. A reviewer who blocked the milder sibling cannot consistently
  wave through the worse one.
- It contradicts criterion-3's own precondition — GC "an `ovd-ns-<slot>` with **no
  live PID**." A *failed read* is not a *confirmed* absence of live PID; the code
  conflates the two.
- The swallow also **hides the catastrophic case** behind silent degradation: a
  cgroup-path regression that makes *every* Running alloc's `cgroup.procs` read
  fail would mass-GC *every* surviving workload netns on *every* restart — and the
  swallow turns that into a silent mass-teardown instead of the loud
  `health.startup.refused` the rest of 04-04 deliberately implements.

**Blast radius** is narrow on a healthy single-node root boot (the survivor's
cgroup scope exists and is readable as root — SPIKE-A), exactly as narrow as the
§5 case the first pass blocked. This is the same "lightest kind of blocking": a
named-rule violation with a real-but-bounded correctness gap — but pointed at a
destructive action.

**Fix (small, well-precedented — mirrors the §5 fix and the `development.md`
worked example):** in `alloc_scope_pids`, distinguish `NotFound` (the cgroup
scope is genuinely gone → the workload's cgroup was reaped → legitimately no live
PID → empty Vec / orphan-eligible) from every other `io::ErrorKind` (EACCES, EIO,
… → **propagate** a typed `NetnsRecoveryError::Observe` → refuse the boot). Apply
the same NotFound-vs-genuine split at the `netns_file_inode` / `read_proc_netns_
inode` sites that feed the orphan classification. This is structurally identical
to the `ChainAbsent`-vs-`TproxyInstall` split `f021b701` just landed, and to the
`read_to_string(proc_filesystems)` `NotFound => v1-signal / _ => propagate`
example in `development.md`. (If the team consciously *wants* fail-toward-GC, that
needs an explicit sanctioned-exception comment justifying why a transient read
failure may delete a live workload's netns — but the default for a destructive
action under uncertainty must be fail-closed, as 04-04 is everywhere else.)
