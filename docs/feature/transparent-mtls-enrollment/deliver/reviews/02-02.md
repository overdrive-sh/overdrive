## issue (blocking) — criterion 4's sysctl prerequisites are NOT verified; one assertion is an outright bug

Criterion 4: *"Spike-proven host prereqs verified present after provision: `ip_forward=1`, `rp_filter` relaxed BOTH globally (`all`+`lo`) AND on the per-host-veth knob (S3 — the two are separate facts/steps), `tx off` on both ends."* The test (`workload_netns_provision.rs:279–299`) contains four sysctl assertions that all pass **independent of whether production's `EnableIpForward` / `RelaxGlobalRpFilter` / `RelaxHostVethRpFilter` converge steps ran at all.**

**(a) Per-host-veth rp_filter assertion is structurally vacuous — a malformed key (outright bug).**
Line 294 builds the key with a `/` separator: `format!("net.ipv4.conf.{}/rp_filter", p.host_veth)`. procps `sysctl` treats `.` and `/` as **swapped** separators, so this key resolves to the bogus proc path `/proc/sys/net/ipv4/conf/<iface>.rp_filter` — which never exists. Ground-truth (Lima, real `ovd-hv-0fff` veth):

```
real knob (dot-form) net.ipv4.conf.ovd-hv-0fff.rp_filter : [2]  exit=0   ← the actual knob
TEST key (slash-form) net.ipv4.conf.ovd-hv-0fff/rp_filter :
  sysctl: cannot stat /proc/sys/net/ipv4/conf/ovd-hv-0fff.rp_filter: No such file or directory  exit=1
```

`sysctl_int` returns `None` on the non-zero exit, so `assert_ne!(None, Some(1))` **always passes**. Proven non-falsifiable: with the real knob forced STRICT (`=1`, simulating a regressed `RelaxHostVethRpFilter`), the slash key still reads `None` → assertion still passes. Production code is *correct* (it writes the dot-form `net.ipv4.conf.{}.rp_filter`); the **test never reads the knob production writes.** This is the precise S3 fact the criterion singles out as "a separate fact/step" — and it is the one fact the test fails to verify.

**(b) The three remaining sysctl assertions are pre-satisfied by the Lima VM defaults → no regression protection.**
Ground-truth VM defaults (no provision run):

```
net.ipv4.ip_forward            = 1     → assert_eq!(_, Some(1)) passes with production removed
net.ipv4.conf.all.rp_filter    = 2     → assert_ne!(_, Some(1)) passes (2 ≠ 1) with production removed
net.ipv4.conf.lo.rp_filter     = 2     → assert_ne!(_, Some(1)) passes with production removed
net.ipv4.conf.default.rp_filter = 2    → fresh veths inherit 2, so even a CORRECT dot-key per-veth
                                          assert_ne!(_, Some(1)) would pass without the relax step
```

Because the VM already ships `ip_forward=1` and `rp_filter=2` (loose) host-globally, every global assertion is satisfied before `provision_workload_netns` is called. A mutation/deletion of the `EnableIpForward` or `RelaxGlobalRpFilter` executor arm would leave the test green. Combined with the **RED_UNIT skip** (no unit test covers the `execute_workload_step → sysctl_set` mapping either), there is **no test anywhere that fails if production stops enabling ip_forward or relaxing rp_filter.** The pure `workload_converge_steps` decision is 02-01-covered, but the executor's sysctl *effect* is covered only by this non-load-bearing Tier-3 assertion.

**Impact.** The step's own acceptance criterion 4 is, in practice, untested for three of its four sysctl facts and *un-testable as written* for the fourth. The GREEN run is real but hollow on this dimension. This is exactly the recurring false-pass class for this feature.

### Required fix (minimal, makes the per-veth fact load-bearing)

The per-host-veth knob is the genuinely per-test-fresh guard (a new veth inherits `default=2`, never `0`), so asserting the **exact value production writes** makes it falsifiable:

```rust
// fix the separator (.) AND assert the exact written value (production writes "0";
// a fresh veth inherits default=2, so a missing relax step now FAILS the assert)
let host_veth_rp = format!("net.ipv4.conf.{}.rp_filter", p.host_veth);
assert_eq!(sysctl_int(&host_veth_rp), Some(0),
    "per-host-veth rp_filter must be relaxed to 0 by RelaxHostVethRpFilter");
```

For the globals: `ip_forward` and `all`/`lo` are host-sticky and shared with the concurrent host-netns suite, so they cannot be cleanly isolated — but assert the **exact** value production writes (`Some(1)` / `Some(0)`) rather than `assert_ne!(_, Some(1))`, and add a one-line comment acknowledging the global knobs are weak guards (the per-veth knob above is the load-bearing rp_filter regression guard). Do NOT deliberately strict-set the globals before provision — that races the concurrent suite.

# RESOLUTION (2026-06-18) — re-review NEEDS_REVISION resolved (test-only, commit `3cba64e0`)

- **Verdict:** **RESOLVED** — the blocking per-host-veth defect is fixed and proven falsifiable; orchestrator independently ground-truthed.
- **Fix commit:** `3cba64e0` (`test(transparent-mtls-enrollment): make 02-02 criterion-4 sysctl asserts load-bearing`) — exactly 1 file changed (`tests/integration/workload_netns_provision.rs`, +33/−11), **no production change**.

**Per-host-veth knob (the blocking finding) — FIXED + load-bearing.** Separator corrected `/` → `.` so the assert reads the knob production writes; tightened to exact `assert_eq!(sysctl_int(&host_veth_rp), Some(0))`. Falsifiable by construction: a fresh veth inherits `default.rp_filter = 2`; the converge always emits `RelaxHostVethRpFilter` (writes `0`) on a (re)built pair, so a regressed step reads `2` and the assert FAILS. Ground-truthed on Lima (fresh veth → `2`; provisioned → `0`; old slash-key → `cannot stat … No such file or directory`).

**Global knobs — the re-review's prescribed `assert_eq!(_, Some(0))` was a verified ERROR; corrected to `!= Some(1)`.** The crafter (correctly, per CLAUDE.md "implement to the design") refused to ship the prescribed global fix because it contradicts production's actual contract — independently confirmed by the orchestrator against the source:
- Production's `sysctl_rp_filter_relaxed` (`veth_provisioner.rs:1882–1884`) treats any value `!= 1` as relaxed, so `workload_converge_steps` emits `RelaxGlobalRpFilter` **only when a global knob is strict (`1`)**. The Lima VM ships `all`/`lo` rp_filter `= 2` (loose) and `ip_forward = 1` — already non-strict — so production **correctly leaves the globals untouched** (ADR-0061: do not rewrite an already-satisfied knob). It therefore never writes `0` to the globals on this kernel; `assert_eq!(_, Some(0))` would FAIL (`left: Some(2), right: Some(0)`). The faithful assertion is `assert_ne!(_, Some(1))` ("not strict"), which mirrors production's own relaxed-predicate. The re-review's own ground-truth table (`all.rp_filter = 2`) is what makes its `Some(0)` prescription self-contradictory.

**Residual (accepted, irreducible — the re-review itself accepted this).** The three global asserts (`ip_forward`, `all`/`lo` rp_filter) remain **weak guards**: VM defaults already satisfy them and the knobs are host-sticky + shared with the concurrent host-netns veth suite, so a regressed `EnableIpForward` / `RelaxGlobalRpFilter` step is not caught. This cannot be made load-bearing without strict-setting the globals before provision, which races the concurrent suite (the re-review explicitly forbade that). The **per-host-veth knob is the load-bearing rp_filter regression guard** and now is. An honest weak-guard comment documents this in the test (no silent cap).

**Non-blocking nitpick (not landed):** `assert_ne!(sysctl_int(global_key), Some(1))` also passes on `None`, so a future malformed *global* key would pass vacuously (the same class as the original per-veth bug). The global keys use the correct dot-form today, so this is latent only; a `assert!(matches!(sysctl_int(k), Some(v) if v != 1))` shape would close it. Not required to resolve the blocking finding.
