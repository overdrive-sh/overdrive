## Findings (priority-ordered)

### `praise:` Atomic claim is genuinely single-lock, and idempotent re-entry precedes the scan

`assign` (689–709) acquires `self.held.lock()` once (695) and performs the held-check (699), the smallest-free scan (704–705) and the insert (706) inside that one guard, dropped at 707 before return — a real single critical section with no contains-then-insert TOCTOU window (`development.md` § "Check-and-act must be atomic"). The idempotent re-entry check (`held.get(&alloc)` → return existing) is placed *before* the scan, so a re-fired `on_alloc_running` returns the existing slot and consumes nothing — even at full capacity (exhaustion gates only NEW claims). C2 is met and is load-bearing: `allocator_re_assign_of_held_alloc_returns_existing_slot_idempotently` (3620–3638) and `allocator_assign_at_capacity_refuses_new_alloc_with_typed_error` (3701–3730) both fail under the obvious mutations.

### `praise:` `unreachable!()` for the in-range `NetSlot::new` and load-bearing tests

`smallest_free_slot` (616–618) uses `.unwrap_or_else(|| unreachable!("0..=NET_SLOT_MAX is in range by construction"))` — the exact idiom `development.md` § "Logically unreachable None/Err" prescribes (not `?`, not `.expect()`). The proptest (3565–3590) asserts **two** orthogonal invariants — chosen slot unheld AND no lower free slot exists — which kills both "return any free" and "return max_held+1 monotonic" mutants in one generator. The at-capacity test asserts three facts (new alloc refused, held alloc still idempotent, error type exact). No vacuous assertions found (the 02-02 lesson held).

### `issue (non-blocking) [C1/C6]:` `snapshot`'s docstring claims a restart-rebuild role it does not have

`snapshot` (726–736) is documented as *"the restart-rebuild / status read surface, criterion 6"* (728–732, and the test docstring at 3732–3736: *"the restart-rebuild ... read surface"*). But criterion 6's rebuild mechanism is **re-assign on each `on_alloc_running`** — which reads nothing from `snapshot`; on a fresh boot `snapshot` returns the *empty* post-restart map. `snapshot` plays **no part** in the rebuild. The mirror-IdentityMgr precedent justifies *having* a read accessor, but the docstring's stated justification is aspirational (`development.md`/CLAUDE.md "No aspirational docs — never document behaviour that is not implemented"). It is also the one method with **zero** consumer that 02-05's rebuild path will *not* wire (assign/release will be; snapshot likely stays a status-only surface). **Fix:** reword the docstring to "a read-only status/observability surface" and drop the "restart-rebuild" claim — or, if it genuinely has no near-term consumer, defer the method until one exists. Non-blocking; does not affect behaviour.

### `nitpick (non-blocking) [C4]:` `NetSlotExhausted` Display can underflow on its own `pub` field

The `#[error(...)]` attribute (590–593) computes `max = capacity - 1`, and `capacity` is a **`pub` field** (597). `NetSlotExhausted { capacity: 0 }.to_string()` underflows — a panic in debug, a `u32::MAX` wrap in release. The only in-tree construction site uses `NET_SLOT_MAX + 1` (≥ 1), so it cannot fire today, but a `pub` error type whose `Display` can panic from a `pub`-settable field is the kind of latent footgun `development.md` § "make invalid states unrepresentable" exists to prevent. **Fix (one line):** `max = capacity.saturating_sub(1)`, or make `capacity` non-`pub` with a constructor. Low severity.

### `question (non-blocking) [C6]:` Is the cross-restart orphan-netns hazard tracked, not just deferred?

Criterion 6 defers cross-restart slot stability as "a tracked follow-up, NOT this step." The concrete hazard the allocator's smallest-free design creates: after a control-plane restart the map is empty and each alloc is re-assigned by smallest-free **in `on_alloc_running` firing order** — so an alloc that held slot 5 may come back as slot 0, while its *old* slot-keyed kernel netns/veth (`ovd-ns-0005`) survived the process restart (kernel objects outlive the CP). The 02-02/02-03 converge-on-boot would then build the new slot's netns and leave the old one orphaned. This is genuinely a 02-05/later concern (no `assign` caller exists yet) and **not an 02-04 defect** — but "tracked follow-up" implies a real issue number. Confirm one exists (per `CLAUDE.md` § "Deferrals require GitHub issues — no hand-wavy forward pointers"); if not, surface it for the user to approve before 02-05 wires the lifecycle hooks.
