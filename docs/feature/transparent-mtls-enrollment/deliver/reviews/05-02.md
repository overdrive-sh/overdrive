## Summary

This is, at its load-bearing core, a genuinely good Tier-3 test. **Oracle 1 + Oracle 2 are the real deliverable, and they are honest.** The netns workload dials a known `service_backends` addr B through the real egress nft-TPROXY redirect to a real `IP_TRANSPARENT` leg-F, the **production** `accept_outbound_and_recover_orig_dst` recovers `orig_dst` via real `getsockname` (`mtls_intercept.rs:868`), the test asserts `recovered == b` AND `recovered.port() != leg_f_port` (`:559-568`), and then feeds the **genuine getsockname output `recovered`** ‚Äî not the test constant `b` ‚Äî into `MtlsResolve.resolve(recovered)` (`:602`), asserting it resolves to `Mesh(ResolvedBackend { addr: B, expected_svid: None })`. That `recovered`-not-`b` feed (`:602`) is the detail that makes the cross-seam single-source claim load-bearing rather than tautological: a wrong recovery would feed a wrong key into resolve and miss the scripted arm. The RED log records a real falsifiability litmus (feed `port+1` ‚Üí RED, `18822 vs 18821`). Capture-then-drop is confirmed by a positive interception signal (`WL-SENT` + real-backend-did-not-accept, `:576-585`) per `debugging.md ¬ß11`, and the Tier-3 hygiene (cross-process `flock`, scrub-at-start-and-end, per-netns resolv.conf dir reaped, `uname -r` recorded, no `--no-run`) is exemplary and matches the 05-01 standard.

The blocking problem is confined to **Oracle 3 (the resolv.conf injection)** and to the **narrative around it**. Oracle 3 does NOT exercise this feature's injection mechanism: it writes the file inline with `std::fs::write` (`:358`) and then asserts the file contains what it just wrote ‚Äî a `fs::write`‚Üí`fs::read` round-trip plus the stock iproute2 per-netns bind-mount convention. Yet the module-doc (`:42-46`) and the assertion messages (`:490`, `:506`) repeatedly claim it proves "the name-layer injection **MECHANISM this feature OWNS** wires." That is precisely the overclaimed/narrated-evidence pattern `.claude/rules/verification.md` classifies as a rejection-class defect ‚Äî and it is doubly avoidable here because the real mechanism is *already* covered, strictly more strongly, by the 02-03 production test (which drives `provision_workload_netns` ‚Üí the private `resolv_conf_write` AND carries a non-vacuous host-negative assertion). ## Issues

### `issue (blocking):` S1 ‚Äî Oracle 3 overclaims: it proves a `fs::write`‚Üí`read` round-trip + the stock bind-mount convention, NOT "the injection mechanism this feature OWNS"

**What the code does.** `inject_resolv_conf` (`:356-360`):
```rust
std::fs::create_dir_all(resolv_conf_dir()).expect(...);
std::fs::write(resolv_conf_path(), resolv_conf_contents(RESPONDER_ADDR)).expect(...);
```
Oracle 3 then reads that same file back (`:485-492`) and asserts it equals `resolv_conf_contents(RESPONDER_ADDR)` (`:488`), plus that `ip netns exec ‚Ä¶ cat /etc/resolv.conf` surfaces the line (`:496-508`).

**What it actually proves**, adversarially:
1. `std::fs::write(p, s)` followed by `std::fs::read_to_string(p)` returns `s` ‚Äî a tautological round-trip of the standard library, traceable to no production logic.
2. The stock `ip netns` per-netns convention bind-mounts `/etc/netns/<ns>/resolv.conf` over `/etc/resolv.conf` inside the namespace ‚Äî a property of **iproute2**, not of this feature's code.

**What it claims to prove.** The module-doc (`:42-46`): "the per-netns ‚Ä¶ resolv.conf (02-03 / D-TME-9, **the injection MECHANISM this feature OWNS**) carries the injected ‚Ä¶ line." The assertion message (`:490`): "(02-03 / D-TME-9 injection mechanism)." The injecting helper's own doc (`:349-355`): "Inject the per-netns resolv.conf **EXACTLY as the 02-03 `veth_provisioner` converge step does**." None of these is true: the production converge step is `resolv_conf_write` (`veth_provisioner.rs:2504`, **private**), reached via `provision_workload_netns`; this test calls neither. The only production symbol it touches is the **pure** `resolv_conf_contents` (`:576`, `pub`), which it does not even import ‚Äî it re-implements it inline (`:260-262`).

**Why this is blocking, not a redundant-but-honest nit.** Per `.claude/rules/verification.md` ¬ß Enforcement: "Narrated evidence ‚Ä¶ 'I'm pretty sure it works like this' is a second assertion by the same entity that made the first ‚Äî reject it." Oracle 3's evidence (the file has the line) is real, but the **claim attached to it** (the feature's injection mechanism wires) is strictly stronger than the evidence supports. The mechanism's correctness rides on `resolv_conf_write` (does it create the dir? write the right path? converge idempotently? leave the host file untouched?), and Oracle 3 tests none of that ‚Äî it tests the test's own `fs::write`. The feature already has the correct, strictly-dominating proof: `provision_injects_node_local_responder_into_netns_resolv_conf` (`workload_netns_provision.rs:432`) drives production `provision_workload_netns` (`:463`), asserts the netns sees the injected line **because the production step ran** (`:480-485`, explicitly load-bearing), AND asserts the host `/etc/resolv.conf` is byte-identical before/after (`:494-499`, non-vacuous host-negative). 05-02 Oracle 3 is a weaker re-statement wearing a stronger claim.

**Resolution (any one closes it):**
- **(preferred)** Reframe Oracle 3's scope honestly in the doc + assertion messages: it proves "the per-netns resolv.conf path convention surfaces an injected `nameserver` line inside the workload's namespace view (the bind-mount plumbing), with the byte-exact D-TME-9 line shape ‚Äî the **production injection mechanism itself is proven by 02-03's `provision_workload_netns` test**, cited here." Drop "the injection MECHANISM this feature OWNS ‚Ä¶ wires" and "EXACTLY as the 02-03 converge step does." This is a comment/string-only change to a test-only commit.
- **(alternative)** Delete Oracle 3 entirely and cite 02-03 as the mechanism proof in the module-doc. Oracle 3 adds no coverage 02-03 lacks; the single-source invariant (the step's actual deliverable) lives in Oracles 1+2, which do not depend on Oracle 3.

Either way the load-bearing claim must stop asserting a mechanism the code does not exercise.

### `suggestion (non-blocking):` S3 ‚Äî the "dependency cycle Cargo rejects" justification is inaccurate; the honest reason is the writer is private

The comment justifying the inline `resolv_conf_contents` duplicate (`:255-259`) states importing the production fn "would form a `worker ‚Üí control-plane ‚Üí worker` dependency cycle Cargo rejects." This is wrong on the mechanism:

1. **Cargo permits dependency cycles that pass through dev-dependencies.** A `overdrive-worker [dev-dependencies] ‚Üí overdrive-control-plane` edge, combined with the existing normal `overdrive-control-plane [dependencies] ‚Üí overdrive-worker` edge (`overdrive-control-plane/Cargo.toml:59`), is a dev-dep cycle ‚Äî Cargo allows these by design (the dev-dep is only built for the depending crate's own tests, breaking the build-order cycle). So importing into the *test tree* would not be rejected on cycle grounds.
2. The genuinely load-bearing reason is different and honest: **`resolv_conf_write` ‚Äî the actual injection writer the criterion is about ‚Äî is private** (`veth_provisioner.rs:2504`, no `pub`), and `resolv_conf_dir`/`resolv_conf_path` are private too (`:2479`,`:2486`). Only the *pure* `resolv_conf_contents` is `pub` (`:576`). So the test cannot drive the production injection from the worker tree regardless of the cycle question ‚Äî the writer isn't reachable.

The "assert the stable observable D-TME-9 wire contract verbatim" half of the justification is fine. Replace the cycle claim with the private-writer fact: "the production writer `resolv_conf_write` is private; only the pure `resolv_conf_contents` is public, so the test asserts the stable D-TME-9 line shape directly. (This is also why S1's mechanism proof belongs to 02-03, which drives the private writer through `provision_workload_netns`.)"
