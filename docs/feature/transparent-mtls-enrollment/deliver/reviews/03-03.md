## issue (blocking): AC2-a over-attributes the agent's host dial reaching the backend to the F5 exemption — the claim is vacuous against the production `iifname` rule, and (for this rule) arguably not even true

**Location:** the AC2-a block (egress_tproxy_capture.rs:594-626), the module docstring
(:450-453), and the `.expect()` message (:608-612):

> `[AC2-a F5 POSITIVE]` … "the agent's HOST-netns dial carrying `SO_MARK =
> MTLS_LEG_S_DIAL_MARK` is **short-circuited by the chain-head exemption** and reaches
> the REAL backend directly (NOT re-captured to leg-F)." (:451-453)
>
> `.expect("F5 positive: the agent's SO_MARK-stamped HOST dial must reach the REAL
> backend directly (**the chain-head exemption short-circuits it before the iifname
> rule**). A timeout means the exemption is broken and the marked dial looped to
> leg-F.")` (:608-612)

**Independently confirmed FALSE against source — I REFUTE the test's causal claim and
CONFIRM the orchestrator's headline.** The production rule installed
(`install_outbound_tproxy` → `run_nft(["…","iifname",host_veth,"meta","l4proto","tcp",
"tproxy",…])`, mtls_intercept.rs:369-388) matches **on the ingress interface**
(`iifname "vethH-egr03"`). The agent's AC2-a dial is a HOST-netns
`dial_with_so_mark(backend, …)` (:600) to a host-`lo` address (`10.200.0.1` is added to
`lo` at setup_topology :269). A host-originated packet to a host-`lo` dst ingresses with
`iif=lo`, never `iif=vethH-egr03`, so the `iifname` rule **structurally cannot match it**
— with or without the F5 exemption. The exemption is therefore *not the reason* the
agent dial reaches the backend; the `iifname` non-match is. The test's own comment at
:623-625 admits this verbatim — "its peer is the loopback source … proving it **never
traversed the veth and was never iifname-matched**" — which is precisely the admission
that the exemption was irrelevant to the outcome the test attributes to it. The
orchestrator's negative control closes the loop empirically: remove the exemption, the
marked host dial STILL reaches the backend (`BACKEND-ACCEPTED peer=10.200.0.1:42948 ->
exemption was NOT load-bearing for this dial`).

**Contrast with the spike, which DID have a load-bearing positive control.** The spike's
egress rule matched on **destination** (`ip daddr 10.200.0.1 tcp dport 18777 tproxy …`,
findings-egress-tproxy.md:68), which DID match the agent's host-`lo` dial — so the
spike's exemption removal genuinely captured the marked dial
(findings-egress-tproxy.md:123-130, *"WITHOUT the exemption it is still captured by
PREROUTING — the `ip route local … table 100` re-injection puts host-originated packets
… through PREROUTING too"*). Production `install_outbound_tproxy` deliberately matches on
`iifname <host_veth>` instead (its own rustdoc, mtls_intercept.rs:295-300: *"NOT the
single-known-backend `ip daddr/tcp dport` shape the egress spike used"*). **Against the
production iifname rule, the "positive" half lost its discriminating power** — and,
because a host-originated leg-S dial to a host-`lo` backend never ingresses the workload
veth, the egress F5 exemption is *not even load-bearing for the agent's leg-S dial on
this rule shape*. The exemption's real job on the **shared** chain is protecting the
inbound `ip daddr/tcp dport` rules (where a host-originated marked dial to a *virt* DOES
match and WOULD loop), not the egress `iifname` rule.

**Why this is blocking (the recurring defect class, not cosmetics).** This is the same
vacuous-/over-attributed-assertion class that bounced 02-02 (always-`None` sysctl keys),
02-03 (vacuous host-unaffected check), and 03-01 (false design citations) to
NEEDS_REVISION in this very feature — and the project rule is unambiguous: *"No
aspirational docs. Never document behaviour that is not implemented"* (CLAUDE.md), and
EDD's enforcement rejects "narrated" / over-claimed evidence
(`.claude/rules/verification.md`). The commit message and the test docstring both assert
AC2-a "proves" the exemption short-circuits the marked dial; roadmap **criterion 2**
names the F5 exemption "spike-proven MANDATORY by the negative control" (ADR-0071 Tier-3
obligation (b)). AC2-b correctly proves the *workload-cannot-self-exempt* half; the
*exemption-is-load-bearing-for-the-agent's-leg-S-dial* half is **unproven**, and for the
`iifname` egress rule **specifically not true** — yet the test name, docstring, and
`.expect()` message all assert it as proven. The next reader (04-01/04-02 wiring leg-S
+ leg-F) will read "AC2-a proves the F5 exemption short-circuits the agent's egress
dial" and build on a false premise about why the exemption exists.

**Minimal remedy (NOT a destructive negative control on the shared VM):** STOP
OVERCLAIMING. Three honest moves, any of which clears the blocker:

1. **Re-document AC2-a truthfully.** Rename the block / fix the docstring + `.expect()`
   to: *"the agent's HOST dial reaches the backend because it originates host-side and
   never ingresses the workload veth, so the `iifname` egress rule cannot match it — this
   path does NOT exercise the F5 exemption (the exemption is irrelevant to the egress
   `iifname` rule; its load-bearing role is on the shared chain's inbound `daddr/dport`
   rules)."* Keep the assertion (it's a true topology fact); fix the *attribution*.
2. **Surface the gap explicitly** in the test docstring and execution log: that the
   egress F5 exemption's load-bearingness for the agent's leg-S dial is a *separate,
   unproven (possibly inapplicable) claim* touching ADR-0071's framing — so the
   ADR-0071 "(b)" obligation is satisfied by AC2-b (self-exempt-impossible) for egress,
   and the positive-control half is deferred/inapplicable. Per CLAUDE.md, if this is a
   real deferral it needs a tracked issue (surface to the user first; do not create
   unilaterally).
3. If the team wants a genuinely load-bearing egress F5 positive control, it must dial
   from a path that **actually ingresses the workload veth** with the leg-S mark (i.e.
   the agent's real leg-S dial path once 04-01/04-02 wire it), not a host-`lo` dial.
   That is correctly out of 03-03's scope — which is exactly why the honest move today
   is (1)+(2), not a fabricated positive.

## suggestion (blocking — fold into the same revision): `accept_outbound_leg`'s blocking `accept()` makes a silent-redirect-failure HANG to the 120 s slow-timeout instead of clean-failing

**Location:** AC1 `accept_outbound_leg(&leg_f, …)` (:548) and AC2-b (:638) drive the
production `leg_f_listener.accept()` (mtls_intercept.rs:723), which is a **blocking
accept with no timeout**. If the egress redirect silently failed (the workload's dial
landed on `backend_fallback` instead of leg-F), AC1's `accept_outbound_leg` blocks
forever → nextest's slow-timeout SIGKILL at 120 s, not an assertion failure. The AC4
and AC2-a paths correctly use `accept_with_timeout` (:490 / :607, 8 s / 5 s bounded);
the two production-driven accepts are the asymmetric exception. For a *walking proof*
whose whole point is "the redirect fires," a hang-to-timeout is a worse failure shape
than a bounded clean-fail (it reads identically to "VM hung" / "test infra broke," per
`.claude/rules/debugging.md` § 11 — an empty/blocked consumer is a downstream symptom,
and the failure mode should name itself). This isn't a correctness defect — the
ground-truth run passed — but it degrades the diagnostic value of a FUTURE regression.
I am tagging it *blocking-fold* (not independently rejection-worthy) only because it
should land in the same revision cycle as the AC2-a fix; the orchestrator may
reasonably downgrade it to non-blocking. **Remedy:** none required if you accept the
hang-to-timeout for a one-shot proof; otherwise, gate the AC1/AC2-b accepts behind the
same bounded poll the other two paths use (the production `accept_outbound_leg` cannot
itself take a timeout without an API change — so the test would need a `set_read_timeout`
/ non-blocking-listener wrapper, or to document the hang-on-failure shape in the
`.expect()` so a future failure is diagnosable).

## thought (non-blocking): topology teardown is imperative end-of-fn, not RAII — a mid-body panic leaks state until the next run's START scrub

`KernelStateLock` and the `TproxyInterceptGuard`/`leg_f` are RAII (release on
unwind), but `teardown_topology()` + `clean_shared_infra()` are imperative calls at the
*end* of the test body (:666-669). If an AC1/AC2-b assertion panics mid-body, those END
calls do not run, and the netns/veth/lo-addr/nft state persists until the next run's
START scrub (:477-478, which tolerates pre-existing). This is mitigated-by-design (the
START scrub is the recovery path, exactly per `.claude/rules/testing.md` § "Leaked …
across runs" — the next run self-heals), and the flock prevents a concurrent run from
tripping over the residue. Acceptable for a single walking proof. A `scopeguard`/`Drop`
wrapper around the topology would make it panic-safe, but that's a refinement, not a
defect — the orchestrator confirmed a clean exit on the passing run.
