## Findings (priority-ordered, Conventional Comments)

### issue (blocking): D1 — the docstring cites "ADR-0071 fact 4" for the iifname-match shape; fact 4 is the `MtlsResolve` enrollment decision, NOT the egress-rule shape. The rule shape is pinned by fact 2.

**Location:** `install_outbound_tproxy` rustdoc, `mtls_intercept.rs:292-293`:

> *"This is the production shape per the feature-delta / ADR-0071 fact 4 — NOT the single-known-backend `ip daddr/tcp dport` shape the egress spike used."*

This is a **factually wrong design citation**, and given this feature's documented "verify unproven claims against the actual evidence" history it must not be rubber-stamped. Read against the SSOT:

- **ADR-0071 fact 4** (`adr-0071…md:260-281`) is titled *"Enrollment per-connection resolve replaces the per-destination map"* — it pins the 3-variant `MtlsResolution` sum type (`Mesh`/`NonMesh`/`MeshUnreachable`) and the `MtlsResolve` driven port. It says **nothing** about `iifname`, the egress rule, or the host-veth match. It is an entirely different decision.
- **ADR-0071 fact 2** (`:247-253`) is the one that pins the egress rule: *"OUTBOUND interception = nft-TPROXY at the host-side veth (the active-side mirror of inbound). The workload's outbound `connect()` leaves its netns via the veth, ingresses the host-side veth, and is nft-TPROXY-redirected to the agent's leg-F `IP_TRANSPARENT` listener."* That is the iifname-on-host-veth-ingress shape — fact **2**, not 4.

So the docstring's load-bearing design anchor points at the wrong fact. The *claim it makes* (that the production shape is iifname-matched, not the spike's daddr/dport shape) is **correct** — it just cites the wrong section to prove it.

**On the substantive correctness question (did the spike validate the iifname shape, or only a daddr/dport shape?):** the spike validated **neither the iifname shape literally**. `findings-egress-tproxy.md:67-68` shows the spike installed:

```
ip daddr 10.200.0.1 tcp dport 18777 tproxy to 127.0.0.1:28777 meta mark set 0x00000001 accept
```

— i.e. the spike used a **`daddr`/`dport`** rule against a single known backend (`10.200.0.1:18777`), NOT an `iifname` rule. The docstring is therefore *honest* that "the egress spike used" the daddr/dport shape. **What the spike actually proved** is the *routing mechanism* — PREROUTING-on-host-veth-ingress + fwmark + `ip rule` + local route + `IP_TRANSPARENT` leg-F + `getsockname` recovery + the F5 exemption — fires for netns egress (`findings:99-114`, with the with/without-TPROXY control). It did NOT prove the **iifname match clause** specifically. That is an honest, design-sanctioned leap, NOT an unproven overreach: ADR-0071 fact 2 + the feature-delta component table (`feature-delta.md:87`: *"nft PREROUTING rule on the host-side veth matching the workload's egress"*) + roadmap criterion 1 (*"matching the workload's egress (ingressing the host veth)"*) all pin the iifname-match as the DESIGN shape, and criterion 5 explicitly assigns the real-fire validation to Tier-3 03-03. So the divergence from the spike's daddr/dport probe to the production iifname rule is **design-sanctioned and correctly deferred** — the docstring's *substance* is right, its *citation* is wrong.

**Why blocking despite being docs-only:** CLAUDE.md § "Implement to the design" binds reviewers to "verify the output against the design's API shape" and the project has a hard rule against propagating wrong references (CLAUDE.md § "Deferrals require GitHub issues — … the next reader treats the deferral as planned work and propagates the false reference"; § "no aspirational docs"). A wrong "fact 4" citation on the single load-bearing design anchor of a production function is exactly the kind of reference that compounds — 03-02/04-01 will read this docstring to understand the egress contract. The 02-01/02-03 reviews bounced docs-only false-premise issues to NEEDS_REVISION for the same reason. **Remedy (one line):** change "ADR-0071 fact 4" → "ADR-0071 fact 2" (the OUTBOUND-interception-at-the-host-veth decision); optionally add "(the iifname-match clause's real fire is the Tier-3 03-03 obligation)" so the spike-vs-production gap is explicit rather than implied.

### issue (blocking): D2 — the idempotency docstrings repeatedly claim the egress rule is "keyed ONLY on host_veth", but every dedup/handle predicate keys on host_veth **AND** port. The claim contradicts the code, and the divergence is a latent duplicate-rule hazard once leg-F's ephemeral port can change for a live veth.

**Location:** three docstrings assert veth-only keying:
- `:304` *"The egress rule is keyed ONLY on `host_veth` (no unique daddr/dport)…"*
- `:327-328` *"the egress rule is keyed only on `host_veth`, so a repeat install for the same veth would stack a literal duplicate."*
- the test-module comment `:991-993` *"Because the rule is keyed ONLY on `host_veth`…"*

But the actual predicates require BOTH conjuncts:
```rust
// dump_has_egress_rule (:635-639)
dump.lines().any(|l| l.contains(&iifname) && l.contains(&redirect))
// find_egress_rule_handle_in_dump (:612-622)
.filter(|l| l.contains(&iifname) && l.contains(&redirect) && l.contains("# handle "))
```
where `redirect = "tproxy to 127.0.0.1:{agent_leg_f_port}"`. So the presence-check is keyed on **`(host_veth, agent_leg_f_port)`**, not `host_veth` alone. The docstring is self-contradicting (the `find_egress_rule_handle_in_dump` rustdoc at `:606-609` states the two-conjunct truth correctly — so the contradiction is *internal* to this one file).

**Is it a real hazard?** Verified the leg-F port lifecycle. `mtls_intercept_worker.rs:191-202` documents leg-F as a **"worker-chosen ephemeral port"** ("the test never has to supply — or even observe — the worker-chosen ephemeral port"); `make_transparent_listener` binds whatever the caller passes, and the worker binds an ephemeral leg-F per alloc. So `agent_leg_f_port` is **NOT node-stable** — it is per-listener-bind. The concrete failure shape:

> `install_outbound_tproxy("ovh-aaaa0", portA)` then, after an agent restart / leg-F re-bind (the 02-06 adopt-on-restart path, where the slot-keyed veth survives but leg-F re-binds a new ephemeral port), `install_outbound_tproxy("ovh-aaaa0", portB)` (same veth, new port) → `dump_has_egress_rule` matches on `(iifname && redirect-to-portB)`, finds the portA rule does NOT carry `:portB`, reads **absent**, and APPENDS a second egress rule → **two TPROXY rules fire for one workload's veth** (the first redirecting to a now-dead portA listener, depending on chain order).

Whether this fires in 03-01's isolation: **no** — 03-01 ships only the install function; nothing re-installs a live veth with a new port yet. The wiring that could trigger it is 04-01 (`start_alloc` installs both rules) + the per-alloc leg-F bind, and the 02-06 adopt-on-restart re-bind. So this is **latent, not currently reachable** — which is exactly why it is a *docs-vs-code* blocker (the claim is wrong today) rather than a behavioural blocker (the divergence cannot yet bite). But the project's own precedent (02-01 B1/B3) is that a latent-but-design-inherited correctness gap with a false reassuring docstring is surfaced, not shipped silent.

**Which is correct — the docstring or the code?** The **code's veth+port keying is the safer choice** and should stay; veth-only keying would mean a re-install with a changed port silently leaves the OLD rule (different failure: stale redirect to a dead listener, never refreshed). The bug is the **docstring**, which describes a weaker key than the code implements and (worse) implies "same veth ⇒ detected as present" when actually "same veth, *different port* ⇒ NOT detected → duplicate appended." **Remedy:** (1) correct all three docstrings to "keyed on `(host_veth, agent_leg_f_port)` — both the ingress interface AND the leg-F redirect target"; (2) state explicitly whether a per-veth re-install with a *changed* leg-F port is expected (if leg-F is node-stable in production, say so and cite where the port is fixed; if it can change — which the 02-06 adopt-on-restart path implies — note the duplicate-on-changed-port hazard as a known 04-01 / 02-06 obligation, with a tracked issue per CLAUDE.md § "Deferrals require GitHub issues" — do NOT leave it as an unstated "keyed ONLY on host_veth" that the next reader trusts). The cheapest honest fix is (1) + a one-line note that 04-01/02-06 must guarantee a stable-per-veth leg-F port (or re-key teardown so a changed port replaces rather than stacks).

### praise: C — the dump-parse predicates are split out pure and the discriminating test genuinely kills both `&&`→`||` mutants on the egress matcher

This is the standout. `find_egress_rule_handle_in_dump` (`:612-622`) and `dump_has_egress_rule` (`:635-639`) are extracted as **pure functions over a `&str` dump**, mirroring the inbound `dump_has_leg_s_exemption` / `ip_rule_dump_has_fwmark` split — so the conjunction logic is unit-killable without a kernel. Traced the mutant kills by hand against the fixtures:

- `egress_rule_requires_iifname_and_redirect_to_match_the_same_rule` (`:1055-1083`) uses a *cross* fixture where line A carries the right iifname (`ovh-aaaa0`) but wrong port (`41999`), and line B carries the right port (`41000`) but wrong iifname (`ovh-other2`). Under correct `&&`, neither line satisfies `(ovh-aaaa0, 41000)` → `false`. Under the `&&`→`||` mutant, line A satisfies the iifname conjunct and line B satisfies the redirect conjunct → mutant returns `true`. The test asserts `!dump_has_egress_rule(cross, "ovh-aaaa0", 41000)` AND `find_egress_rule_handle_in_dump(cross, …) == None` — so it **kills the `||` mutant on BOTH predicates** in one fixture. This is the genuine no-single-line-satisfies-both construction, not a tautology.
- `egress_predicate_does_not_mistake_an_inbound_daddr_rule_for_an_egress_rule` (`:1102-1117`) pins that an inbound `ip daddr`/`tcp dport` rule sharing the `tproxy to 127.0.0.1:41000` tail is NOT read as an egress rule — killing the over-broad-needle mutant. That directly defends the inbound/egress coexistence the shared-chain model requires.
- `egress_handle_parsed_per_host_veth_in_a_multi_rule_chain` (`:1085-1100`) pins distinct handles (7, 12) for distinct veths → the by-handle Drop deletes exactly its own rule.

The fixture `EGRESS_CHAIN_DUMP` is a verbatim-shaped real `nft -a` capture (quoted iifname, zero-padded `0x00000001` set-mark, trailing `# handle <N>`), and the `0x00000002` exemption matches the verified `MTLS_LEG_S_DIAL_MARK = 0x2`. No vacuous assertions in the egress suite — every assertion would flip under a wrong predicate. The 02-02/02-03 vacuity lesson held here.

### issue (non-blocking): T1 — the third conjunct of `find_egress_rule_handle_in_dump` (`l.contains("# handle ")`) has no mutant-killing test; dropping it is a surviving mutant.

**Location:** `find_egress_rule_handle_in_dump:620` — `.filter(|l| l.contains(&iifname) && l.contains(&redirect) && l.contains("# handle "))`.

The discriminating test kills the iifname↔redirect pair, but the **third conjunct (`"# handle "`) is not independently exercised**. Every line in every egress fixture that carries `iifname && redirect` ALSO carries `# handle <N>`, so a mutant that drops `&& l.contains("# handle ")` (or flips it to `||`) still passes every test — the subsequent `.find_map(parse_handle)` returns `None` for a handle-less line anyway, so behaviour is *coincidentally* preserved. Criterion 6 (`≥80%`) would likely still clear (the killed `&&` + needle mutants dominate), but the `# handle ` conjunct is a genuine surviving mutant. Note the inbound twin `find_virt_rule_handle` (`:572-580`) carries the *identical* untested third conjunct — pre-existing pattern faithfully mirrored, not a regression. **Remedy (optional):** add a fixture line that matches `iifname && redirect` but has NO `# handle ` marker and assert `find_egress_rule_handle_in_dump(...) == None` for it.

### thought (non-blocking): F — `install_outbound_tproxy` reads the chain dump TWICE for the same predicate, and the step-2 pair re-scans the same dump twice.

**Location:** `install_outbound_tproxy:330-372`. Two observations, both low-severity given the converge-on-boot single-node shell-out context:

1. **Redundant same-dump re-scan (step 2).** `dump_has_egress_rule(&dump, …) && let Some(existing) = find_egress_rule_handle_in_dump(&dump, …)` walks the same `dump` twice for the *same* `(iifname && redirect)` predicate. Since `find_egress_rule_handle_in_dump` returns `None` exactly when `dump_has_egress_rule` returns `false` (for a well-formed dump where the matching line carries `# handle `), the `dump_has_egress_rule &&` guard is **fully redundant** — `if let Some(existing) = find_egress_rule_handle_in_dump(&dump, host_veth, agent_leg_f_port)` alone is equivalent and reads the dump once. The only divergent case (line matching `iifname && redirect` but lacking `# handle `) falls through to append in BOTH versions — behaviour-preserving. **Remedy (optional):** drop the `dump_has_egress_rule(&dump, …) &&` conjunct in step 2; keep `dump_has_egress_rule` (still needed as the standalone unit-tested predicate).
2. **TOCTOU between step 2 (presence check) and step 4 (handle recover) — NOT a real concern.** A second concurrent install for the same veth could interleave between step 2 and step 3 and both append (the classic check-then-act TOCTOU). BUT: (a) the inbound twin has the identical shape and the inbound Tier-3 test serialises concurrent installs with a cross-process `flock` (`mtls_intercept_install.rs:75-99` `KernelStateLock`) — so the concurrency hazard is a known, mitigated property of the whole shared-chain model, not new to egress; (b) in production `install_outbound_tproxy` is driven once per alloc from the action-shim StartAllocation arm, not concurrent for the same veth. The step-4 second `list_chain()` is unavoidable (the kernel assigns the handle on append; you can only learn it by re-dumping) and matches the inbound `find_virt_rule_handle` shape exactly.

### praise: E — `install_outbound_tproxy` correctly handles handle-recovery failure with a typed error and `ok_or_else`, where the inbound twin does it inline.

`install_outbound_tproxy` (`:366-372`) wraps `find_egress_rule_handle(...)?` (returning `Result<Option<u64>>`) with `.ok_or_else(|| InterceptError::TproxyInstall { reason: format!("could not recover nft rule handle for egress host_veth {host_veth} → 127.0.0.1:{agent_leg_f_port} after append") })?` — a cause-naming typed error per `.claude/rules/development.md` § Errors, with the exact `host_veth`/port for operator diagnostics. The egress split (pure `…_in_dump` → `Option`, thin `…_handle` → `Result<Option>`, install → `ok_or_else` to `Result`) is a *better* factoring than the inbound `find_virt_rule_handle` (which embeds the not-found case inline), and the `Ok(None)`="not present yet" / `Err`="couldn't read the chain" distinction is honest. No silent absorb, no `unwrap`, no `expect`.

## Disposition

**NEEDS_REVISION.** The **code behaviour and tests are sound** — the rule shape is design-correct (iifname-on-host-veth per ADR-0071 fact 2 + feature-delta + roadmap), the shared-infra reuse is genuinely verbatim with zero invented public surface, the by-handle teardown + typed-error handle-recovery are clean, and the pure dump-parse predicates are extracted and **genuinely mutation-killed** by a real no-single-line-satisfies-both discriminating fixture (the standout praise). The default-lane tests were orchestrator-ground-truthed at 13/13.

The verdict is driven by **two docs-only blocking defects that are load-bearing and propagate**: **D1** cites "ADR-0071 fact 4" (the `MtlsResolve` enrollment decision) where it means fact 2 (the OUTBOUND-at-host-veth decision) — a wrong design anchor on the function's single design-conformance claim, in a feature with a documented history of misciting spike/design evidence; and **D2** asserts "keyed ONLY on host_veth" in three places while the code keys on `(host_veth, agent_leg_f_port)` — a self-contradiction (the `find_egress_rule_handle_in_dump` rustdoc in the same file states the two-conjunct truth correctly) that masks a real latent duplicate-egress-rule hazard once leg-F's verified-ephemeral port can change for a live veth (reachable at 04-01 / the 02-06 adopt-on-restart re-bind, not in 03-01's isolation). Both fixes are documentation edits — change the cite to fact 2; correct the three keying claims to `(host_veth, port)` and state the changed-port re-install expectation (with a tracked issue if leg-F is not node-stable). The non-blocking items (T1 untested third conjunct; D3 comment-skip vs exclude_re; D4 misleading headline-test name + orchestration-is-Tier-3 acknowledgement; F redundant same-dump re-scan + benign TOCTOU note) should fold into the same revision.
