## Findings (priority-ordered)

### Blocking
None. No design-API divergence, no invented trait surface, gates green.

### praise: Faithful to the contract SSOT over the loose roadmap text
Roadmap criterion 2 ("scripted-fault mode returns `MeshUnreachable` / a typed store-fail
error") predates the precise 01-01 contract. The crafter correctly modeled `MeshUnreachable`
as a scriptable **`Ok` arm** (via the table) and `StoreUnreadable` as the **only** `Err` —
the exact asymmetry 01-01's rustdoc pins. Right reading of "implement to the design."

### praise: Lock + TOCTOU discipline explicit and correct
`resolve` takes the armed fault into a local (`let armed_fault = self.resolve_fault.lock().take();`)
before branching, with a comment naming why (guard dropped at statement end). No lock across a
branch; no `.await` inside `resolve`.

### praise: Scripting style matches the `SimMtlsEnforcement` precedent
`script_resolve_fault(&self, …)` mirrors the sibling adapter's `script_trip` / `mark_stalled`
one-shot, consumed-on-use, `&self`-interior-mutability pattern; required ctor params, no
production-binding default; does not over-constrain `expected_svid` (the `None` discipline is
correctly left to the 01-03 host adapter).

### question (non-blocking): One-shot `resolve_fault` vs the contract's idempotency invariant
01-01 states `resolve` is idempotent in observable state for the same `orig_dst` against an
unchanged backend set. The one-shot fault means `arm → Err → Ok` for two consecutive same-addr
calls. A *transient* store fault that clears is arguably faithful, but the
`mtls_resolve_equivalence` harness (05-02) must arm the host adapter's `StoreUnreadable` with
matching one-shot-then-recover semantics — the host's poisoned-handle/corrupt-table fault is
naturally *persistent*. **Carry this forward to the equivalence author.**

### suggestion (non-blocking): Constrain or document the fully-general `default_arm`
`default_arm: MtlsResolution` accepts any arm, but the host adapter can only ever return
`NonMesh` for an unscripted `orig_dst`. A harness scripting `default_arm = Mesh(…)` would emit
an outcome the host structurally cannot — latent equivalence drift. Add a one-line rustdoc note
that the equivalence harness MUST pass `NonMesh` to preserve host-faithfulness.

### question (non-blocking): `with_probe_failure(self)` vs `script_resolve_fault(&self)` asymmetry
Probe failure uses an owned builder; the resolve fault uses `&self` interior mutability.
Defensible (probe is a startup property; the resolve fault is armed mid-test), just
stylistically inconsistent with the sibling precedent, which scripts everything via `&self`.
Fine as-is.

### nitpick (non-blocking): `RED_ACCEPTANCE` log hygiene
01-02 logged `RED_ACCEPTANCE: EXECUTED/PASS`, but this pure default-lane sim adapter has no
port-to-port acceptance scaffold — 01-01 correctly logged the same phase `SKIPPED /
NOT_APPLICABLE`. Cosmetic; correct next time the log is touched.

### nitpick (non-blocking): `#[allow]` vs `#[expect]` on the test module
`#[allow(clippy::expect_used, clippy::unwrap_used)]` is correct and idiomatic for test code
(`warn`-level lints). Optional: the project leans toward self-removing `#[expect(…)]` where
applicable.

## Forward note to DELIVER (05-02 `mtls_resolve_equivalence`)

The host (`ServiceBackendsResolve`, 01-03) and sim must agree on `StoreUnreadable` fault
semantics. The sim's one-shot `script_resolve_fault` clears after one `resolve`; ensure the
host-side store fault is armed with matching transient-then-recover behavior, or the
equivalence assertion will diverge.
