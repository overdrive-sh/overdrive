# Verifier-budget baseline — xdp_service_map_lookup
#
# Slice 04 / S-2.2-14 — post-Maglev MAGLEV_MAP-indexed slot lookup
# baseline. Slice 07's `cargo xtask verifier-regress` gate (lands at
# step 07-01) parses this file and fails the PR if the recorded
# program exceeds the baseline by >20% or approaches the per-program
# complexity ceiling (1,000,000 instructions for CAP_BPF programs)
# by >10%, per ASR-2.2-03.
#
# History:
#   Slice 02 baseline (commit 8edf471): verified_insns = 401 with
#     flat HashMap<ServiceKey, BackendEntry> single-lookup shape.
#   Slice 03 baseline (commit c38a110): verified_insns = 460 with
#     HASH_OF_MAPS chained lookup (outer SERVICE_MAP → inner ARRAY)
#     plus a placeholder slot hash (`(src_port ^ dst_port) & 0xff`).
#     Growth: +59 instructions (+14.7%) — within ASR-2.2-03's 20%
#     delta gate vs Slice 02.
#   Slice 04 (this commit): verified_insns = 660 with FNV-1a-32 over
#     the 5-tuple unrolled across 13 input bytes plus
#     `% MaglevTableSize::DEFAULT` (= 16_381) for slot derivation,
#     and the inner ARRAY populated by `maglev::permutation::generate`
#     instead of round-robin. The HoM swap orchestration (5-step,
#     ADR-0040 § 2) is unchanged — only the inner-ARRAY contents and
#     the kernel-side slot-hash differ.
#
#     Delta vs Slice 03: 460 → 660 (+200 instructions, +43.5%).
#
#     Delta vs Slice 02 baseline: 401 → 660 (+259 instructions, +64.6%).
#     This is a one-time architectural bump from FNV-1a 5-tuple hashing.
#     Slice 04 is the SECOND baseline recorded after Slice 02; the
#     +64.6% delta is informational vs the original baseline, NOT
#     gating against it. The next gate (Slice 07's verifier-regress)
#     enforces against THIS new baseline (660), with ASR-2.2-03's 20%
#     delta becoming `insns ≤ 792` for any future change.
#
#     660 is comfortably within the 50% CAP_BPF target (500_000 insns)
#     — the program uses 0.13% of the L1-cache-fits budget. There is
#     no further FNV/Maglev-shaped growth queued; subsequent slices
#     (DROP_COUNTER PerCpuArray, TC egress, sockops mTLS) live in
#     separate programs with their own baselines.
#
# Recorded against:
#   kernel:   6.8.0-111-generic   (Ubuntu 24.04 LTS lineage; matrix row "6.6 LTS"
#                                   per .claude/rules/testing.md § "Kernel matrix"
#                                   with Lima's actual 6.8 kernel)
#   commit:   post-Maglev (commit f2b4210; step 04-04)
#   date:     2026-05-06
#   tool:     aya 0.13.1 ProgramInfo::verified_instruction_count
#             (veristat is not packaged in the Lima image; aya queries
#             BPF_OBJ_GET_INFO_BY_FD which returns the same kernel-side

#   target:   target/xtask/bpf-objects/overdrive_bpf.o

#             rustup nightly + bpfel-unknown-none + -Z build-std=core,

#
# Budget gates (ADR-0040 / architecture.md §7 D3 / ASR-2.2-03):
#   verified_insns ≤ 792        ← +20% delta vs Slice 04 baseline (660)
#                                 — pinned by the integration test
#                                 `verifier_budget_xdp_service_map_lookup_within_20pct_of_baseline`
#                                 in `crates/overdrive-dataplane/tests/integration/atomic_swap.rs`.
#   verified_insns ≤ 500,000    ← 50% of CAP_BPF ceiling, "L1-cache fits"
#   verified_insns ≤ 600,000    ← 60% absolute ceiling per ASR-2.2-03
#   verified_insns ≤ 1,000,000  ← absolute CAP_BPF ceiling (kernel 5.10+)
#
# Recorded value (Slice 04):
#   verified_insns = 660
#   660 / 500,000 = 0.13% of the L1-cache-fits target. Comfortably within budget.
#
# DST companion:
#   `MaglevDeterministic` invariant
#   (`crates/overdrive-sim/src/invariants/maglev_deterministic.rs`) pins
#   the K3 twin-run identity property of `maglev::generate` — two
#   calls with identical inputs return bit-identical `Vec<BackendId>`

#   the inner-ARRAY contents whose verifier complexity this baseline
#   measures; non-determinism in `generate` would produce a different
